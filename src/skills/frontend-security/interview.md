# 前端安全面试题

## Q1. XSS 是什么？为什么危险？

**答：**

XSS 是跨站脚本攻击，本质上是攻击者把恶意脚本注入页面，并让浏览器把它当成正常脚本执行。

它危险的原因在于：

- 一旦脚本执行成功，就可能读取页面数据
- 可以发起同源请求
- 可以窃取可读存储中的信息
- 可以篡改页面和冒用用户操作

所以 XSS 不只是“弹个 alert”，而是真正能威胁用户身份和数据的一类攻击。

---

## Q2. 存储型、反射型、DOM 型 XSS 的区别是什么？

**答：**

### 存储型 XSS

恶意内容被存进数据库，后续其他用户访问页面时触发。

### 反射型 XSS

恶意输入随着请求进入响应并被立即执行。

### DOM 型 XSS

问题主要发生在前端脚本里，是前端把不可信数据直接插入 DOM 导致执行。

现代前端项目里，DOM 型 XSS 很常见，因为很多问题来自：

- `innerHTML`
- URL 参数拼接
- 富文本展示

---

## Q3. 如何防 XSS？

**答：**

常见防护方式包括：

- 输出转义
- 不使用危险 DOM API
- 富文本白名单过滤
- 使用 CSP
- 关键 Cookie 配 `HttpOnly`
- 限制第三方脚本来源

如果要一句话总结：

- 不把不可信字符串当 HTML 或脚本执行

---

## Q4. CSP 是什么？有什么作用？

**答：**

CSP 是内容安全策略，用于告诉浏览器：

- 哪些脚本可以执行
- 哪些资源可以加载
- 哪些来源是可信的

它的作用主要是：

- 即使有注入点，也进一步限制恶意内容执行

所以 CSP 通常是对抗 XSS 的重要补充防线，而不是替代转义本身。

---

## Q5. CSRF 是什么？为什么会发生？

**答：**

CSRF 是跨站请求伪造，本质上是攻击者利用浏览器自动携带身份凭证的机制，诱导用户在不知情的情况下向目标站点发起请求。

它之所以会发生，通常是因为：

- 浏览器会自动带 Cookie
- 服务端只认 Cookie，不校验请求是否来自合法页面

所以核心问题不是“请求是不是发得出去”，而是：

- 这个请求是不是用户真实意愿

---

## Q6. 如何防 CSRF？

**答：**

常见方式：

- `SameSite` Cookie
- CSRF Token
- `Origin` / `Referer` 校验
- 关键操作二次确认

面试里最好补一句：

- `HttpOnly` 主要防 XSS 读 Cookie，本身不是 CSRF 的核心防线

---

## Q7. `HttpOnly`、`Secure`、`SameSite` 分别做什么？

**答：**

- `HttpOnly`：禁止 JavaScript 读取 Cookie
- `Secure`：只在 HTTPS 下发送 Cookie
- `SameSite`：限制跨站请求是否自动携带 Cookie

这三个属性都很重要，但解决的问题不同，不能混为一谈。

---

## Q8. 前端权限控制为什么不能代替后端授权？

**答：**

因为前端代码运行在用户环境里，攻击者完全可以绕过前端，直接构造请求访问后端接口。

前端权限控制更多是：

- 控制界面展示
- 改善用户体验

真正的授权判断必须在后端完成。

一句话：

- 前端决定“看不看得到”
- 后端决定“做不做得到”

---

## Q9. Token 放 `localStorage` 和放 Cookie 各有什么风险？

**答：**

### 放 `localStorage`

优点是简单，但风险是：

- 一旦 XSS，容易被脚本读走

### 放 Cookie

如果配合 `HttpOnly`，JS 不能直接读，但要额外注意：

- CSRF 防护

所以没有绝对完美答案，关键是看整体认证方案和威胁模型。

---

## Q10. CORS 是安全机制吗？

**答：**

它是浏览器的跨源访问控制机制，但不能把它理解成完整安全方案。

它主要控制的是：

- 浏览器是否允许前端脚本读取跨域响应

它不能代替：

- 身份认证
- 权限校验
- 接口防滥用

所以面试里更准确的说法是：

- CORS 是浏览器访问控制机制，不是后端鉴权机制

---

## Q11. 点击劫持是什么？怎么防？

**答：**

点击劫持通常是攻击者把目标页面放进透明 iframe，诱导用户点击，实际触发危险操作。

常见防护：

- `X-Frame-Options`
- CSP 的 `frame-ancestors`

现代更推荐使用：

- `frame-ancestors`

因为它更灵活。

---

## Q12. 为什么说前端输入校验不是安全边界？

**答：**

因为攻击者可以完全绕过前端页面，直接请求后端接口。

前端校验的主要价值是：

- 提升交互体验
- 提前发现明显错误

真正决定安全性的仍然是：

- 服务端校验

所以前端校验很有必要，但不能当成真正的安全防线。
