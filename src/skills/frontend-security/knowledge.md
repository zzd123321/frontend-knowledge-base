# 前端安全核心知识点

前端安全不是“会背 XSS 和 CSRF”就够了。真正做项目时，安全问题往往出现在多个层的交界处：

- 用户输入进入 DOM
- 登录态如何存储和传递
- 跨域与 Cookie 如何配合
- 第三方脚本是否可信
- 权限控制是否只停留在前端

所以前端安全的核心，不是“前端单独把所有问题解决”，而是清楚前端能防什么、不能防什么，以及应该和后端、网关、运维共同完成哪些防线。

---

## 1. 前端安全到底在保护什么

前端安全关注的目标通常包括：

- 用户身份
- 用户数据
- 页面内容完整性
- 业务操作可信性
- 浏览器运行环境安全性

常见风险包括：

- 恶意脚本注入
- 越权访问
- 登录态被窃取
- 恶意请求伪造
- 第三方资源被篡改

安全的本质不是“绝对不出事”，而是：

- 尽量减少攻击面
- 提高攻击成本
- 降低单点失守后的损失

---

## 2. 前端安全的边界

前端不能独立解决所有安全问题。

前端能做的：

- 减少 XSS 注入点
- 正确使用 Cookie、Token
- 做输出转义和安全渲染
- 限制危险操作入口
- 配合安全头和浏览器策略

前端做不到的：

- 真正的权限裁决
- 防止客户端源码被看到
- 防止用户伪造本地请求
- 防止攻击者直接绕过前端调后端接口

所以一个非常重要的认知是：

- 前端可以参与安全，但前端不能代替服务端安全

---

## 3. 常见安全问题地图

前端最常见的安全主题包括：

- XSS
- CSRF
- 点击劫持
- 敏感信息泄露
- Token / Cookie 使用不当
- 权限控制误解
- 第三方脚本风险
- CORS 误配置
- 文件上传与富文本风险
- Prompt Injection 风险（AI 应用场景）

理解这张地图，比死背单个定义更重要。

---

## 4. XSS 是什么

XSS 是 Cross-Site Scripting，跨站脚本攻击。

本质是：

- 攻击者把恶意脚本注入页面
- 浏览器把它当成正常脚本执行

后果可能包括：

- 读取页面数据
- 冒用用户身份发请求
- 篡改页面内容
- 窃取可读存储中的敏感信息

对前端来说，XSS 是最核心、也最常见的一类风险。

---

## 5. XSS 为什么危险

因为一旦恶意脚本在你的页面上下文里执行，它就拥有和正常前端代码类似的能力。

例如它可以：

- 读 `localStorage`
- 读非 `HttpOnly` Cookie
- 获取表单输入
- 发起同源请求
- 修改 DOM
- 劫持用户操作

很多人误以为 XSS 只是“弹个 alert”，但真实风险远比这个严重。

---

## 6. XSS 的常见类型

### 存储型 XSS

恶意内容被存进数据库，后续被其他用户访问时触发。

### 反射型 XSS

恶意输入进入请求后立即在响应中反射出来。

### DOM 型 XSS

问题不一定来自服务端返回，而是前端脚本自己把不可信内容直接插入 DOM。

在现代前端项目里，DOM 型 XSS 尤其需要警惕，因为很多问题发生在：

- `innerHTML`
- 富文本渲染
- URL 参数拼接
- 第三方内容注入

---

## 7. 常见 XSS 注入点

最危险的注入点通常包括：

- `innerHTML`
- `outerHTML`
- `insertAdjacentHTML`
- `document.write`
- 动态创建 `script`
- URL 拼接到 HTML 中
- 样式和事件属性拼接

例如：

```js
el.innerHTML = userInput
```

如果 `userInput` 不可信，就很危险。

---

## 8. 如何防 XSS

常见防护方式：

- 输出转义
- 富文本白名单过滤
- 不用危险 DOM API
- 使用 CSP
- Cookie 配合 `HttpOnly`
- 限制第三方脚本来源

对前端最直接的原则是：

- 不把不可信字符串当 HTML 执行

---

## 9. 输出转义为什么重要

很多场景下，用户输入并不危险，危险的是你怎么输出它。

例如同样一段字符串：

- 当作纯文本显示，通常安全
- 当作 HTML 解析，就可能危险

所以很多安全问题的关键不是“输入有问题”，而是：

- 输出上下文错误

例如 HTML、属性、URL、样式、脚本上下文的处理方式并不相同。

---

## 10. 富文本为什么是高风险区

富文本场景经常需要允许一部分 HTML，但这正好是安全难点。

风险包括：

- `script` 标签
- 事件属性如 `onclick`
- `javascript:` URL
- 恶意内联样式
- 嵌入第三方危险资源

因此富文本不能直接“信任后展示”，通常需要：

- 严格白名单
- 过滤危险标签和属性
- 必要时服务端和前端双重清洗

---

## 11. CSP 是什么

CSP 是 Content Security Policy，内容安全策略。

它可以通过响应头或 meta 标签告诉浏览器：

- 哪些脚本允许执行
- 哪些样式允许加载
- 哪些资源来源是可信的

它的价值在于：

- 即使有些注入点漏了，也能额外限制恶意脚本执行

所以 CSP 是对抗 XSS 的重要第二道防线。

---

## 12. CSP 常见策略思路

常见方向包括：

- 限制脚本来源为 `'self'`
- 禁止内联脚本
- 限制样式来源
- 限制图片、字体、接口来源
- 限制 frame 嵌入来源

关键认知：

- CSP 不是替代转义
- 它是额外保护层

如果站点大量依赖内联脚本、内联样式、任意第三方资源，CSP 会很难配好。

---

## 13. `HttpOnly` 的价值

`HttpOnly` 是 Cookie 的一个重要属性。

作用是：

- 禁止 JavaScript 读取 Cookie

它不能防止请求自动携带 Cookie，但可以降低：

- XSS 直接窃取登录态的风险

所以会话型 Cookie 一般很推荐配合：

- `HttpOnly`
- `Secure`
- `SameSite`

---

## 14. CSRF 是什么

CSRF 是 Cross-Site Request Forgery，跨站请求伪造。

本质是：

- 攻击者诱导已登录用户访问恶意页面
- 浏览器自动带上用户对目标站点的身份凭证
- 恶意页面借此发起本不该发出的请求

它和 XSS 不同。

XSS 是：

- 恶意脚本执行在受害页面里

CSRF 是：

- 利用浏览器自动携带身份信息的机制

---

## 15. 为什么会发生 CSRF

最核心原因通常是：

- 浏览器会自动携带 Cookie

如果服务端只认：

- 这个 Cookie 有效

却不额外验证请求来源或 CSRF Token，那么攻击者就可能借用户身份发请求。

所以 CSRF 的关键点不是“能不能请求接口”，而是：

- 请求是不是用户真实意愿

---

## 16. 如何防 CSRF

常见方式：

- `SameSite` Cookie
- CSRF Token
- `Origin` / `Referer` 校验
- 关键操作二次确认

其中最常见组合是：

- 会话 Cookie + CSRF Token

或：

- 严格的 `SameSite` 策略 + 来源校验

---

## 17. `SameSite` 的作用

`SameSite` 用于限制跨站请求是否自动携带 Cookie。

常见值：

- `Strict`
- `Lax`
- `None`

理解重点：

- `Strict` 最严格
- `Lax` 更平衡
- `None` 允许跨站，但必须配合 `Secure`

它对防 CSRF 很重要，但不能代替所有校验逻辑。

---

## 18. 点击劫持是什么

点击劫持通常指：

- 攻击者把目标网站页面放进透明 iframe
- 诱导用户点击
- 实际点到的是隐藏的危险按钮

例如：

- 转账
- 授权
- 删除操作

这个问题的防护重点在于：

- 限制页面被第三方 iframe 嵌入

---

## 19. 如何防点击劫持

常见手段：

- `X-Frame-Options`
- CSP 中的 `frame-ancestors`

现代更推荐：

- `frame-ancestors`

因为它更灵活，也更符合统一 CSP 管理思路。

---

## 20. CORS 和安全是什么关系

CORS 经常被误解成“安全机制本身”。

更准确地说：

- CORS 是浏览器的跨源访问控制机制
- 它主要限制前端脚本读取跨域响应

它不能解决：

- 身份认证
- 权限控制
- 接口是否允许别人直接调用

所以：

- CORS 配得再严，也不能代替后端鉴权

---

## 21. 认证与授权的区别

这两个概念必须分清。

### 认证 Authentication

确认“你是谁”。

### 授权 Authorization

确认“你能做什么”。

前端常做的很多事情，比如：

- 根据角色隐藏按钮
- 根据权限展示菜单

更多是体验层的权限控制，不是最终授权裁决。

真正授权仍然必须由后端控制。

---

## 22. Token、Cookie、Session 的安全差异

### Cookie

浏览器可自动携带，适合会话管理，但要关注：

- CSRF
- `HttpOnly`
- `SameSite`
- `Secure`

### Token

通常手动放进 `Authorization` 头，更灵活，但要关注：

- 存储位置
- XSS 风险
- 刷新机制

### Session

通常保存在服务端，浏览器侧一般持有 Session ID。

所以它们不是完全并列的同一级概念。

---

## 23. Token 存哪更安全

这是前端里非常高频的问题。

没有绝对完美答案，但要理解取舍。

### 放 `localStorage`

优点：

- 简单
- 容易读取和附带

风险：

- 一旦 XSS，容易被读走

### 放 Cookie

优点：

- 可配 `HttpOnly`
- JS 无法直接读取

风险：

- 需要防 CSRF

所以关键不是机械背“放哪最安全”，而是：

- 你要结合整体认证方案和威胁模型做选择

---

## 24. JWT 的常见误区

很多人误以为：

- JWT 就天然安全
- JWT 不需要服务端状态

实际上：

- JWT 只是 Token 的一种格式
- 它不是天然加密，只是可签名校验
- 失效、撤销、续期都需要完整设计

JWT 适合某些场景，但不是所有系统都应该默认上 JWT。

---

## 25. 前端权限控制的正确定位

前端权限控制主要价值在于：

- 改善用户体验
- 减少无意义操作入口
- 让界面和角色一致

但它不能代替服务端权限校验。

因此正确认知是：

- 前端控制“看不看得到”
- 后端控制“做不做得到”

只在前端做权限判断是危险的。

---

## 26. 敏感信息泄露

前端很容易不小心泄露这些信息：

- 源码里的密钥
- 调试日志里的用户隐私
- 错误上报中的敏感参数
- URL 上的 token
- Source Map 中的实现细节

非常重要的原则：

- 浏览器能拿到的内容，最终都不能被当成真正秘密

所以不要把真正机密直接放进前端代码或公开环境变量。

---

## 27. Source Map 的安全问题

Source Map 对调试非常有用，但如果线上公开暴露，也可能带来风险：

- 源码结构暴露
- 内部接口路径暴露
- 注释和业务逻辑暴露

所以更常见的安全做法是：

- 生成 Source Map 用于错误平台上传
- 但不要直接对公网开放

---

## 28. 第三方脚本风险

前端项目经常接第三方：

- 统计脚本
- 广告脚本
- 聊天插件
- 可视化埋点

风险在于：

- 它们运行在你的页面上下文里
- 拥有和你自己脚本近似的能力

所以第三方脚本本质上是供应链风险的一部分。

---

## 29. SRI 是什么

SRI 是 Subresource Integrity。

作用是：

- 给外部资源加哈希校验
- 浏览器只在资源内容匹配时才执行

适合：

- 加载 CDN 静态库

它能降低某些第三方资源被篡改后的风险。

---

## 30. 文件上传为什么也是安全问题

文件上传不仅是功能问题，也有安全风险。

常见问题：

- 上传恶意脚本文件
- 伪造文件类型
- 超大文件攻击
- 图片马
- 上传后可执行资源暴露

前端能做的包括：

- 限制类型和大小
- 做基础校验

但真正安全控制仍然需要服务端完成。

---

## 31. 前端输入校验有什么意义

前端输入校验当然有意义，但它主要是：

- 改善交互体验
- 提前拦截明显错误

它不能被当成真正安全边界，因为：

- 攻击者可以绕过前端直接请求后端

所以正确理解是：

- 前端校验是体验优化
- 后端校验才是安全边界

---

## 32. DOMPurify 这类工具的意义

针对富文本和 HTML 清洗场景，常会使用专门的 sanitizer。

这类工具的价值是：

- 用白名单方式清理危险标签和属性
- 降低自己手写过滤逻辑的风险

但要注意：

- 工具不是银弹
- 配置和上下文同样重要

---

## 33. iframe 的安全控制

iframe 本身不是不安全，但如果使用不当，问题很多。

常见控制手段：

- `sandbox`
- `allow`
- `referrerpolicy`
- 限制嵌入来源

尤其在嵌入第三方内容时，应该尽量：

- 最小化权限
- 不给不必要能力

---

## 34. 前端安全与 AI 场景

如果项目接入 AI，还会出现新的前端相关风险：

- Prompt Injection 展示污染
- 模型输出中的恶意链接
- AI 生成 HTML 直接渲染
- 对话日志中的隐私泄露

因此 AI 输出也不能默认信任。

例如：

- AI 返回的 Markdown / HTML 仍然要做安全渲染

---

## 35. 前端安全测试与检查

常见检查方式：

- 代码审查
- ESLint / 安全规则
- 依赖漏洞扫描
- CSP 报告
- 渗透测试
- 富文本和输入点专项检查

安全问题很少靠“某一个工具”完全解决，更多是：

- 开发规范 + 自动检查 + 关键点人工评审

---

## 36. 常见安全误区

- 以为只要用了 HTTPS 就安全了
- 以为前端权限控制就等于授权
- 以为 CORS 能代替鉴权
- 以为输入校验做了就够了
- 以为 JWT 天然比 Session 安全
- 以为 `localStorage` 只是“方便一点”
- 以为 XSS 只是弹窗问题

这些误区比不会某个 API 更危险。

---

## 37. 前端安全最值得掌握的主线

如果只抓一条主线，就抓这条：

1. 识别不可信输入从哪里来
2. 明白哪些地方会把不可信内容执行起来
3. 理解认证、授权、Cookie、Token 的边界
4. 用浏览器安全机制和服务端策略共同收缩攻击面
5. 不把前端体验层控制误当成真正安全边界

把这条主线建立起来，前端安全就不再是一堆名词，而会变成一套能真正指导开发决策的判断框架。
