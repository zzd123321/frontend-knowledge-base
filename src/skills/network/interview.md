# 网络与 HTTP 面试题

## Q1. 从输入 URL 到页面显示，中间发生了什么？

**答：**

这是一道非常经典的综合题，面试官通常不是要你背流程，而是想看你对浏览器、网络和渲染链路有没有整体理解。

一个比较完整但不过度细碎的回答通常可以分成以下几个阶段：

### 1. URL 解析

浏览器首先会解析输入内容，判断它是一个 URL、搜索关键词，还是本地地址。如果是 URL，就会继续解析出：

- 协议
- 域名
- 端口
- 路径
- 查询参数

### 2. DNS 解析

浏览器需要把域名解析为 IP 地址。这个过程通常会依次查找：

- 浏览器缓存
- 操作系统缓存
- 本地 DNS 服务器
- 更上游的 DNS 服务器

### 3. 建立连接

如果是 HTTP，一般会建立 TCP 连接；如果是 HTTPS，还会在 TCP 之上继续做 TLS 握手，用于加密和证书校验。

### 4. 发送 HTTP 请求

浏览器会构造请求报文，包括：

- 请求行
- 请求头
- 请求体

然后发送给服务器。

### 5. 服务器处理并返回响应

服务端接收请求后，会进行路由、业务处理、数据库访问、模板渲染或接口返回，最终返回 HTTP 响应。

### 6. 浏览器解析响应

如果响应是 HTML，浏览器会开始：

- 解析 HTML，构建 DOM
- 解析 CSS，构建 CSSOM
- 执行 JS
- 构建渲染树
- 布局、绘制、合成

### 7. 页面显示与后续资源加载

页面不是“最后一下同时出现”的，而是逐步加载与渲染。  
CSS、JS、图片、字体、异步请求可能继续影响页面后续表现。

如果要再进一步加分，可以补一句：

- 过程中还可能涉及缓存命中、重定向、CDN、Service Worker、懒加载等机制

## Q2. TCP 和 UDP 的区别是什么？

**答：**

TCP 和 UDP 都属于传输层协议，但设计目标完全不同。

### TCP

TCP 是面向连接、可靠传输的协议。  
它的特点包括：

- 建立连接后再传输
- 保证数据按顺序到达
- 丢包会重传
- 有流量控制和拥塞控制

因此 TCP 更适合对可靠性要求高的场景，比如：

- HTTP / HTTPS
- 文件传输
- 数据库连接

### UDP

UDP 是无连接的，设计更轻量。  
它的特点包括：

- 不保证可靠到达
- 不保证顺序
- 不做复杂重传控制
- 协议开销更小

因此 UDP 更适合对实时性更敏感、能容忍少量丢包的场景，比如：

- 实时音视频
- 游戏
- DNS 查询

这题最重要的不是背“TCP可靠、UDP不可靠”，而是说明：TCP 和 UDP 不是谁替代谁，而是面向不同目标设计的两类协议。

## Q3. 为什么 TCP 要三次握手？

**答：**

三次握手的核心目的，是让通信双方确认彼此的收发能力正常，并建立可靠连接。

可以简化理解为：

1. 客户端发送连接请求，说明“我能发”
2. 服务端响应，说明“我收到了你的请求，而且我也能发”
3. 客户端再次确认，说明“我也收到了你的响应”

为什么不能两次？

因为两次无法让双方都确认“双方都具备收发能力”。  
三次握手本质上是在确认：

- 你能收到我的消息
- 我也能收到你的消息
- 双方的发送和接收链路都是通的

面试时不建议只机械背 SYN、ACK，更建议把它讲成“建立可靠连接前的双向确认过程”。

## Q4. 常见 HTTP 方法有哪些？`PUT` 和 `PATCH` 有什么区别？

**答：**

常见 HTTP 方法包括：

- `GET`
- `POST`
- `PUT`
- `PATCH`
- `DELETE`
- `OPTIONS`
- `HEAD`

### 常见语义

- `GET`：获取资源
- `POST`：提交数据、触发创建或业务动作
- `PUT`：整体更新资源
- `PATCH`：部分更新资源
- `DELETE`：删除资源

### `PUT` 和 `PATCH` 的区别

理论语义上：

- `PUT` 更偏向“整体替换”
- `PATCH` 更偏向“局部修改”

比如一个用户对象原来有 10 个字段：

- `PUT` 常理解为提交完整对象后替换原对象
- `PATCH` 常理解为只传改动字段

但也要强调一点：HTTP 方法的语义是协议建议，真正是否严格遵守，还要看后端实现。

## Q5. 常见 HTTP 状态码有哪些？

**答：**

面试里不需要把所有状态码背完，但要掌握常用高频状态码的语义。

### 2xx 成功

- `200 OK`：请求成功
- `201 Created`：资源创建成功
- `204 No Content`：请求成功但没有返回体

### 3xx 重定向

- `301 Moved Permanently`：永久重定向
- `302 Found`：临时重定向
- `304 Not Modified`：协商缓存命中，资源未修改

### 4xx 客户端错误

- `400 Bad Request`：请求参数或格式有问题
- `401 Unauthorized`：未认证或认证失效
- `403 Forbidden`：已认证但无权限
- `404 Not Found`：资源不存在
- `405 Method Not Allowed`：请求方法不被允许
- `409 Conflict`：资源状态冲突

### 5xx 服务端错误

- `500 Internal Server Error`
- `502 Bad Gateway`
- `503 Service Unavailable`
- `504 Gateway Timeout`

如果面试官追问 `401` 和 `403` 的区别，一定要说清楚：

- `401` 偏向“你还没通过身份认证”
- `403` 偏向“你是谁我知道，但你没有权限”

## Q6. `GET` 和 `POST` 的区别是什么？

**答：**

这道题最容易被答浅。  
正确回答应该区分“语义层面”和“常见实现差异”。

### 语义层面

- `GET`：用于获取资源
- `POST`：用于提交数据或触发处理

### 常见实现差异

- `GET` 参数通常出现在 URL 中
- `POST` 数据通常放在请求体里
- `GET` 更容易被缓存、收藏、分享
- `POST` 一般不用于直接书签保存

### 常见误区

很多人会说：

- “GET 不安全，POST 安全”
- “GET 只能传少量数据”

这些说法都不够严谨。

更准确的说法是：

- 真正是否安全主要取决于是否使用 HTTPS、服务端校验是否充分
- URL 长度限制更多是浏览器和服务器实现限制，不是 HTTP 协议硬性规定

这题的关键是：不要把“习惯差异”说成“绝对本质”。

## Q7. HTTP 为什么是无状态的？怎么解决？

**答：**

HTTP 无状态是指：协议本身不会自动记住前后两次请求之间的业务上下文。  
服务端默认不会因为你上一次访问过，就天然知道这一次还是不是你。

为什么会这么设计？

- 协议简单
- 易于扩展
- 服务器更容易横向扩展

但业务场景里我们又经常需要“记住用户登录状态”，因此就出现了会话保持机制，比如：

- Cookie
- Session
- Token
- JWT

所以更准确的理解是：

- HTTP 协议本身无状态
- 业务层通过额外机制补上“状态”

## Q8. Cookie、Session、Token 的区别是什么？

**答：**

这题很经典，建议从“存在哪里”和“谁负责维护”两个维度来答。

### Cookie

Cookie 是浏览器保存的一小段数据，会在符合规则时自动随着请求发送给服务端。

它本身不是会话方案，而是一种客户端存储与自动携带机制。

### Session

Session 通常指服务端保存的一份会话数据。  
浏览器只保存一个会话标识（通常在 Cookie 中），服务端根据这个标识找到对应会话。

### Token

Token 是服务端签发给客户端的一段凭证，客户端后续主动在请求头中携带它，服务端据此识别身份。

### 关系理解

- Cookie：侧重“浏览器如何存和发”
- Session：侧重“服务端如何维护用户会话”
- Token：侧重“客户端携带身份凭证”

所以它们不是完全并列的概念。  
真实项目里常见组合是：

- Cookie + Session
- Token + Authorization Header

## Q9. 什么是 HTTPS？它解决了什么问题？

**答：**

HTTPS 本质上是 HTTP 运行在 TLS 之上。  
它解决的核心问题有三个：

### 1. 机密性

通过加密，防止中间人窃听传输内容。

### 2. 完整性

防止传输过程中内容被篡改。

### 3. 身份认证

通过证书让客户端确认自己连接的服务器身份可信，而不是被冒充的网站。

如果只使用 HTTP，攻击者可能做的事情包括：

- 偷看请求内容
- 篡改页面返回内容
- 伪装成目标网站

所以 HTTPS 的价值不只是“加密一下”，而是建立可信通信链路。

## Q10. 强缓存和协商缓存有什么区别？

**答：**

浏览器缓存主要分为两类：强缓存和协商缓存。

### 强缓存

如果命中强缓存，浏览器直接使用本地缓存资源，甚至不会发请求到服务器。

常见相关头：

- `Cache-Control`
- `Expires`

### 协商缓存

如果资源需要验证，浏览器会发请求询问服务端“这个资源有没有变化”。  
如果服务端确认没变，就返回 `304 Not Modified`，浏览器继续使用本地缓存。

常见相关头：

- `ETag` / `If-None-Match`
- `Last-Modified` / `If-Modified-Since`

一句话总结：

- 强缓存：直接用
- 协商缓存：先问服务器再决定用不用

---

## Q11. `no-cache` 和 `no-store` 的区别是什么？

**答：**

这是非常高频的细节题，很多人容易答反。

### `no-cache`

它不是“不缓存”，而是“可以缓存，但每次使用前必须先向服务器验证是否过期”。

### `no-store`

它才是真正的“不允许缓存”，浏览器和中间代理都不应该存储响应内容。

所以区别在于：

- `no-cache`：能存，但要先验证
- `no-store`：不存

面试时如果能把这个点讲准确，说明你对缓存语义不是停留在字面理解。

---

## Q12. `ETag` 和 `Last-Modified` 的区别是什么？

**答：**

二者都用于协商缓存，但依据不同。

### `Last-Modified`

基于资源最后修改时间。  
优点是简单，缺点是精度有限，比如短时间内多次修改可能识别不准确，而且某些内容没变但时间戳变了也可能造成误判。

### `ETag`

基于资源内容特征生成标识。  
优点是更精确，缺点是生成和比对可能比时间戳成本稍高。

在实际项目中，很多系统会同时使用两者：

- `Last-Modified` 作为基础兜底
- `ETag` 提供更精确判断

## Q13. 什么是跨域？是服务器限制还是浏览器限制？

**答：**

跨域本质上是浏览器同源策略带来的限制。  
“同源”要求协议、域名、端口三者一致。

因此：

- 前端页面在 `https://a.com`
- 请求 `https://api.a.com`

这在浏览器眼里就可能已经是跨域。

为什么说这是浏览器限制而不是服务器限制？

因为如果你用后端去请求，很多时候并不存在所谓“跨域”问题。  
跨域主要是浏览器为了安全，限制脚本随意读取不同源资源。

所以要把这题答准，关键是说清楚：跨域是浏览器安全模型问题，不是 HTTP 协议本身的限制。

## Q14. 什么是 CORS？预检请求是什么？

**答：**

CORS 是跨源资源共享机制，是现代浏览器处理跨域资源访问的标准方案。

它的核心思路是：浏览器默认不允许跨域读数据，但如果目标服务端明确通过响应头声明“我允许这个源访问”，浏览器就可以放行。

常见响应头包括：

- `Access-Control-Allow-Origin`
- `Access-Control-Allow-Methods`
- `Access-Control-Allow-Headers`
- `Access-Control-Allow-Credentials`

### 预检请求

对于某些“复杂请求”，浏览器不会直接发正式请求，而是先发一个 `OPTIONS` 请求去问服务端：“这个跨域请求你允不允许？”

如果服务端返回允许，浏览器才继续发正式请求。  
这一步就叫预检请求。

所以预检请求本质上是浏览器的安全确认机制，而不是业务方主动多发的一次请求。

---

## Q15. 什么情况会触发预检请求？

**答：**

常见触发条件包括：

### 1. 请求方法不是简单方法

比如：

- `PUT`
- `DELETE`
- `PATCH`

### 2. 使用了自定义请求头

比如：

- `X-Token`
- 自定义业务头

### 3. `Content-Type` 不是简单类型

例如：

- `application/json`

为什么很多前端一发 JSON 请求就预检？就是因为 `application/json` 不属于简单请求头的安全白名单。

这题如果能补充“预检失败时正式请求根本不会发出去”，就更完整。

## Q16. JSONP 为什么只支持 GET？

**答：**

JSONP 的原理是利用 `<script>` 标签不受同源策略限制这一点，动态创建一个 script 标签去请求跨域资源。

而 `<script src="...">` 本质上只能发 GET 请求，所以 JSONP 也天然只能支持 GET。

JSONP 的优点是兼容老浏览器和老系统；缺点也很明显：

- 只支持 GET
- 需要服务端配合特殊格式
- 有安全和维护成本

因此现代项目里 JSONP 基本已经被 CORS 取代。

## Q17. HTTP/1.1、HTTP/2、HTTP/3 的主要区别是什么？

**答：**

这题不要只说“HTTP/2 更快”，要说明为什么。

### HTTP/1.1

特点：

- 支持长连接
- 但请求在一个连接里容易受队头阻塞影响
- 头部信息重复发送较多

### HTTP/2

核心改进：

- 二进制分帧
- 多路复用
- 头部压缩

多路复用的意义在于：多个请求可以在一个连接中并行传输，减少了过去“一个连接一个请求顺序阻塞”的问题。

### HTTP/3

HTTP/3 基于 QUIC，而 QUIC 底层使用 UDP。  
它进一步改善了连接建立速度和某些队头阻塞问题。

所以这题的关键不是背版本差异，而是说明：协议演进的核心目标是更低延迟、更高传输效率、更好的网络适应性。

## Q18. WebSocket 和 HTTP 轮询的区别是什么？

**答：**

### HTTP 轮询

客户端每隔一段时间主动发一次请求询问服务端有没有新数据。

问题在于：

- 请求开销大
- 延迟高
- 空请求多

### WebSocket

WebSocket 建立后是长连接，支持全双工通信。

这意味着：

- 服务端可以主动推送消息给客户端
- 客户端也可以随时发消息
- 更适合实时场景

比如：

- 聊天
- 协同编辑
- 实时行情

所以更准确的说法是：轮询是一种“不断问”，WebSocket 是一种“连上后持续双向交流”。

## Q19. 前端如何做接口请求性能优化？

**答：**

这题建议按网络链路和业务层面两个方向回答。

### 网络层优化

- 用缓存减少重复请求
- 开启压缩
- 使用 CDN
- 使用 HTTP/2 或更现代协议

### 请求层优化

- 避免无意义重复请求
- 请求去重
- 合并请求或按需拆分
- 取消失效请求
- 预取关键数据

### 资源和页面层优化

- 懒加载
- 首屏只请求必要资源
- 控制接口瀑布流依赖

如果是中高级面试，我还会补一句：性能优化不能只看单次请求时长，还要关注请求数量、依赖关系、缓存命中率和主线程是否被请求回调逻辑拖慢。

## Q20. 什么是 CSRF？常见防护方式有哪些？

**答：**

CSRF 是跨站请求伪造。  
简单理解就是：攻击者诱导一个已经登录目标网站的用户，在不知情的情况下发起一个本不该发出的请求。

为什么会发生？因为浏览器可能会自动携带用户在目标站点的 Cookie。

常见防护方式包括：

### 1. SameSite Cookie

限制跨站请求自动携带 Cookie。

### 2. CSRF Token

服务端生成随机 token，前端提交时必须带上，攻击者难以伪造。

### 3. Origin / Referer 校验

服务端检查请求来源是否合法。

### 4. 关键操作二次确认

例如支付、转账等场景加入额外确认。

面试里最好顺带补一句：CSRF 和 XSS 不是一回事，CSRF 是利用用户身份，XSS 是注入恶意脚本。

## Q21. 浏览器里看到 `304 Not Modified`、`from memory cache`、`from disk cache` 分别代表什么？

**答：**

这题本质上是在考你是否真的理解浏览器缓存链路，而不是只会背“强缓存和协商缓存”。

### `304 Not Modified`

表示浏览器发起了请求，服务端确认资源没有变化，于是返回 `304`，浏览器继续使用本地缓存。

这说明：

- 发生了网络请求
- 走的是协商缓存
- 服务器参与了校验

### `from memory cache`

表示浏览器直接从内存缓存中取资源，通常不会真正发请求到服务器。

特点：

- 速度最快
- 常见于同一页面生命周期内重复使用资源

### `from disk cache`

表示浏览器从磁盘缓存读取资源，也通常不会真正发起网络请求。

特点：

- 比内存缓存慢一点
- 但依然远快于重新下载

一句话总结：

- `304`：发了请求，再决定继续用旧资源
- `from memory cache` / `from disk cache`：直接用本地缓存，通常不发请求

## Q22. 为什么前端一发 `application/json` 跨域请求，经常就先出现一次 `OPTIONS`？

**答：**

因为这通常不再属于“简单请求”，浏览器会先发预检请求。

预检的目的不是拿业务数据，而是先确认：

- 这个源能不能跨域访问
- 这个方法能不能用
- 这个请求头能不能带

很多人会误以为“前端多发了一次无用请求”，其实这是浏览器安全模型的一部分。

之所以 `application/json` 很容易触发预检，是因为：

- 简单请求允许的 `Content-Type` 范围有限
- `application/json` 不属于那组简单类型

所以浏览器会先发 `OPTIONS`，服务端只有明确返回允许信息后，正式请求才会继续发出。

如果预检失败：

- 正式请求不会继续发
- 控制台常会看到 CORS 报错

---

## Q23. 为什么明明接口状态码是 `200`，前端还是报跨域错误？

**答：**

这是一个很常见也很容易误判的问题。

原因在于：跨域问题很多时候不是“服务器没响应”，而是“浏览器不允许前端脚本读取这个响应”。

也就是说：

- 请求可能已经到达服务端
- 服务端也可能已经正常返回 `200`
- 但如果响应头不满足 CORS 要求，浏览器仍然会拦截结果

典型原因包括：

- 缺少 `Access-Control-Allow-Origin`
- 前端带了凭证，但服务端用了 `*`
- 预检请求没有正确返回允许头
- 实际请求和预检允许的方法或请求头不匹配

所以看到这类问题时，不要只看业务接口日志，还要看：

- `OPTIONS` 预检是否成功
- Response Headers 里有没有正确的 CORS 头
- 前端是否设置了 `credentials`

## Q24. 前端怎么定位“接口慢”到底慢在哪里？

**答：**

这题很适合拉开“会用接口”和“会排查问题”的差距。

可以从浏览器 Network 面板的 Timing 维度回答。

常见阶段包括：

- Queueing / Stalled：请求排队等待
- DNS Lookup：DNS 解析耗时
- Initial connection：TCP 建连耗时
- SSL：TLS 握手耗时
- Request sent：请求发送耗时
- Waiting for server response：服务端处理和首字节返回耗时
- Content Download：响应下载耗时

不同阶段慢，排查方向完全不同：

- DNS 慢：域名解析、网络环境、第三方域过多
- Connection / SSL 慢：建连成本高、弱网、握手开销大
- Waiting 慢：后端处理慢、网关慢、接口依赖链长
- Download 慢：响应体过大、带宽差、压缩没开

如果要再加分，可以补一句：

- 不要只盯单个请求耗时，还要看是否存在请求瀑布流、重复请求和主线程阻塞

## Q25. WebSocket 和 SSE 应该怎么选？

**答：**

二者都可以做实时通信，但能力边界不同。

### WebSocket

特点：

- 长连接
- 全双工
- 客户端和服务端都能主动发消息

适合：

- 聊天
- 协同编辑
- 实时互动

### SSE

特点：

- 基于 HTTP
- 服务端到客户端单向推送
- 使用相对简单

适合：

- 日志流
- 任务进度
- 通知流

怎么选：

- 如果需要双向实时通信，优先 WebSocket
- 如果只是服务端持续推送文本事件流，SSE 往往更轻

所以不是谁更高级，而是看通信模型是否匹配业务。
