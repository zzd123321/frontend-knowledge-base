# Git 面试题

## Q1. Git 的工作区、暂存区、本地仓库分别是什么？

**答：**

这是 Git 最基础但也最重要的模型题。

### 工作区

就是你当前正在编辑文件的目录，也就是磁盘上真实可见的项目文件。

### 暂存区

可以理解成“下一次提交的候选清单”。  
你通过 `git add` 把某些改动放进暂存区，Git 会根据暂存区内容来生成下一次提交。

### 本地仓库

是你本地已经提交保存下来的版本历史。

这三者的关系非常重要，因为 Git 的很多操作本质上都在处理这三层之间的数据流转。

---

## Q2. `git add` 的作用是什么？

**答：**

`git add` 的作用不是提交，而是把工作区改动加入暂存区。

它的意义很大，因为 Git 允许你把同一次编辑中的不同改动拆成不同提交。  
也就是说，Git 的提交不是“把当前所有改动一股脑提交”，而是“把暂存区里的内容提交”。

这也是为什么 Git 能支持更细粒度、更干净的提交历史。

---

## Q3. `git fetch` 和 `git pull` 有什么区别？

**答：**

### `git fetch`

只把远程最新历史同步到本地的远程跟踪分支，比如 `origin/main`，但不会自动把这些变化合并到你当前分支。

### `git pull`

通常等于：

```bash
git fetch + git merge
```

也就是说它会在拉取后直接尝试合并到当前分支。

所以更成熟的协作习惯通常是：

- 先 `fetch`
- 再看差异
- 再决定 merge / rebase

因为这样更可控。

---

## Q4. `merge` 和 `rebase` 有什么区别？

**答：**

这是 Git 高频核心题。

### `merge`

把一个分支的提交历史合并到当前分支，并保留原有分叉结构。  
它的优点是：

- 不改写历史
- 历史更真实

缺点是历史图有时会更复杂。

### `rebase`

把当前分支的提交“挪到”另一个分支最新提交之后，让历史看起来更线性。

优点：

- 历史更整洁

缺点：

- 会改写提交历史

所以成熟实践一般是：

- 私有开发分支可适度 rebase 整理历史
- 共享分支不要轻易 rebase 后强推

---

## Q5. 为什么 Git 需要暂存区？

**答：**

因为暂存区让 Git 的提交粒度变得非常灵活。

没有暂存区的话，你一次提交基本就只能提交当前所有改动。  
有了暂存区之后，你可以：

- 只提交当前功能相关文件
- 把一个大改动拆成多个小提交
- 先整理提交内容再 commit

这对于：

- code review
- 回滚
- 历史可读性

都非常重要。

所以暂存区不是“多此一举”，而是 Git 能够支持高质量提交历史的关键设计。

## Q6. `git reset`、`git revert`、`git restore` 的区别是什么？

**答：**

这三个命令都和“撤销”有关，但作用层面不同。

### `git restore`

更偏文件层面的恢复，比如恢复工作区文件内容，或者把文件从暂存区拿出来。

### `git reset`

更偏引用和状态层面的回退，它会移动分支指针，具体影响范围取决于：

- `--soft`
- `--mixed`
- `--hard`

### `git revert`

不会改写已有历史，而是新增一个“反向提交”去抵消某次旧提交的效果。

所以更安全的共享历史撤销方式通常是 `revert`，而不是随意 `reset` 后强推。

---

## Q7. 什么是冲突？怎么解决？

**答：**

冲突是指 Git 在合并两段历史时，无法自动判断某一部分内容该保留哪一边。

常见出现场景：

- merge
- rebase
- cherry-pick

解决流程通常是：

1. 打开冲突文件
2. 阅读冲突标记
3. 手动整理出最终正确内容
4. `git add`
5. 继续 merge/rebase

这题如果想答得成熟一点，可以补一句：解决冲突不是“把标记删掉”就结束了，真正重要的是你是否理解两边业务改动的意图。

---

## Q8. 什么是 `stash`？适合什么场景？

**答：**

`stash` 可以把当前未提交的改动临时保存起来，让工作区回到相对干净状态。

适合场景包括：

- 当前功能做到一半，临时要去修另一个 bug
- 需要切分支但当前改动还不想 commit

它的本质可以理解成一个“临时改动收纳区”，不是长期替代提交的方式。

所以比较成熟的习惯是：短期切上下文可以 stash，长期保留最好还是形成清晰提交。

---

## Q9. `cherry-pick` 的作用是什么？

**答：**

`cherry-pick` 的作用是把某个指定提交单独应用到当前分支。

它适合：

- 把某个 bugfix 从一个分支移植到另一个分支
- 不想整条分支都合并，只想摘取其中一个提交

所以它本质上是“选择性搬运提交”，而不是整分支合并工具。

---

## Q10. 什么是 HEAD？

**答：**

HEAD 可以理解成 Git 当前所在位置的引用。

大多数时候，HEAD 指向当前分支；当前分支再指向最新提交。  
所以你可以把它理解成“你现在正在基于哪个提交工作”。

这题看似简单，但在理解：

- checkout / switch
- detached HEAD
- reset / rebase

这些操作时都很关键。

---

## Q11. 什么是远程跟踪分支？

**答：**

像 `origin/main` 这样的分支，通常叫远程跟踪分支。  
它表示的是“上一次同步时，远程仓库这个分支在本地的映射状态”。

它不是你当前真正开发的本地分支，而是一个本地记录，用来帮助你比较：

- 远程和本地差了多少
- 需要 pull 还是 push

这题的关键是别把 `origin/main` 和本地 `main` 混为一谈。

---

## Q12. 为什么不建议在共享分支上强推？

**答：**

因为强推会改写远程历史。  
一旦别人已经基于旧历史开发，强推后就可能出现：

- 历史分叉
- 本地无法顺利同步
- 提交丢失或难以对齐

所以共享分支上强推是高风险操作。  
除非团队明确约定且所有人知情，否则不建议随意这么做。

---

## Q13. `git reflog` 有什么用？

**答：**

`reflog` 用于查看 HEAD 和引用的变更记录。  
它非常适合处理“后悔药”类场景，比如：

- reset 过头了
- rebase 搞乱了
- 误删提交了

很多表面上“找不到了”的提交，其实都可以通过 `reflog` 找回来。

所以这题很能体现你有没有真正处理过 Git 事故。

---

## Q14. Tag 的作用是什么？

**答：**

Tag 一般用于给某个特定提交打上版本标记，比如：

- `v1.0.0`
- `release-2026-04`

它的典型用途包括：

- 发布版本标识
- 快速定位历史发布点
- 回滚时找到准确版本

所以 tag 的本质是“给某个历史提交起一个更稳定、更有业务语义的名字”。

---

## Q15. `.gitignore` 为什么有时不生效？

**答：**

这是很典型的细节题。  
`.gitignore` 只对“还没有被 Git 跟踪的文件”生效。

如果某个文件已经被 Git 跟踪了，即使你后来把它写进 `.gitignore`，Git 也不会自动停止跟踪它。

所以这题的关键点是：

- `.gitignore` 管的是“是否忽略未跟踪文件”
- 不是“把已经纳管的文件自动移出版本控制”

---

## Q16. 什么是分布式版本控制？

**答：**

分布式版本控制的核心含义是：每个开发者本地都有完整仓库历史，而不是只依赖中心服务器才能查看历史和提交。

这带来的好处包括：

- 本地就能查看历史
- 本地就能提交
- 更适合离线工作
- 分支操作更灵活

Git 相比集中式版本控制系统，一个很大的优势就在这里。

---

## Q17. 如何理解一个好的 commit？

**答：**

一个好的 commit 不只是“代码能跑”，而应该满足：

- 单一职责
- 改动自洽
- 信息清楚
- 易于 review
- 易于回滚

如果一个 commit 同时包含：

- 新功能
- 样式重构
- 配置调整
- Bug 修复

那 review 和回滚都会变得很痛苦。

所以好的 commit 本质上是在维护团队协作质量。

---

## Q18. 什么时候用 `revert`，什么时候用 `reset`？

**答：**

这题一定要结合“是否共享历史”来答。

### 用 `revert`

当某次错误提交已经进入共享历史时，更安全的做法通常是 `revert`，因为它不会改写已有历史。

### 用 `reset`

当你只是本地整理未共享历史时，`reset` 更直接。

所以可以总结成：

- 共享历史：优先 `revert`
- 本地整理：可用 `reset`

---

## Q19. 为什么很多团队要求 feature branch 工作流？

**答：**

因为它能够把不同开发任务隔离开，减少直接污染主分支的风险。

它的价值包括：

- 每个功能独立开发
- 更方便发 PR / MR 做 code review
- 主分支更稳定
- 回滚和追踪更容易

所以 feature branch 本质上不是形式主义，而是团队协作效率和稳定性的基础。

---

## Q20. Git 协作中最重要的习惯有哪些？

**答：**

我会总结为以下几条：

- 小步提交
- 提交前先看 diff
- 不随意改写共享历史
- 合并前先同步主分支最新状态
- 冲突解决后认真验证
- 提交信息表达清晰

这些习惯看起来基础，但真正决定了一个团队的 Git 协作成本。
