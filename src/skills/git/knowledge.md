# Git 核心知识点

Git 不只是“提交代码”的工具，它本质上是团队协作、历史追踪、问题定位、分支管理和发布回滚的基础设施。前端开发里，很多协作效率问题、回滚问题、代码 review 问题，最后其实都和 Git 使用方式有关。

学 Git 最重要的不是背命令，而是建立正确的心智模型：

- 当前改动在哪一层
- 这个命令会改变哪一层
- 这个操作会不会改写历史
- 这段历史是否已经共享给别人

把这四件事想清楚，很多 Git 问题都会简单很多。

---

## 1. Git 是什么

Git 是分布式版本控制系统。

它的核心能力包括：

- 记录文件历史
- 管理版本差异
- 支持分支并行开发
- 支持多人协作同步
- 允许回滚和追踪问题来源

和“只保存当前文件状态”的备份方式不同，Git 记录的是：

- 某一时刻项目整体快照
- 快照之间的演变关系

---

## 2. Git 与 SVN 的差异

Git 和传统集中式版本控制的关键区别在于：

- Git 本地就保存了完整仓库历史
- 分支和合并更轻量
- 离线也能提交和查看历史

这带来的好处：

- 开发灵活
- 分支试验成本低
- 本地回溯能力强

因此 Git 更适合现代团队并行协作。

---

## 3. Git 的三棵树思维

理解 Git 最常用的一套模型是“三棵树”：

- 工作区 Working Directory
- 暂存区 Staging Area / Index
- 本地仓库 Repository

### 工作区

你当前正在编辑的真实文件。

### 暂存区

准备进入下一次提交的内容集合。

### 本地仓库

已经提交保存下来的历史。

很多 Git 命令之所以让人混乱，就是因为它们会操作这三层里的不同对象。

---

## 4. Git 的基本工作流

最经典流程是：

1. 修改工作区文件
2. `git add` 把选定改动放进暂存区
3. `git commit` 把暂存区生成提交
4. `git push` 推到远程

用代码表示：

```bash
git add src/App.vue
git commit -m "fix: handle empty state"
git push origin feature/empty-state
```

这条链路看起来简单，但真正关键的是：

- `add` 并不是提交
- `commit` 也不是推送

每一步职责不同。

---

## 5. 文件状态

Git 里常见文件状态：

- `untracked`
- `modified`
- `staged`
- `committed`

查看状态最常用：

```bash
git status
```

理解这些状态非常重要，因为你要知道：

- 哪些文件 Git 还没管
- 哪些改动还在工作区
- 哪些已经进入下一次提交

---

## 6. 为什么暂存区很重要

很多初学者觉得暂存区只是多余一步，其实不是。

暂存区的价值在于：

- 允许你控制提交粒度
- 允许把一次大改动拆成几个独立提交
- 让提交历史更干净、更有语义

例如你同时做了两件事：

- 修复登录 bug
- 顺手改了一个按钮样式

通过暂存区，你可以把它们拆成两个提交，而不是混在一起。

---

## 7. 提交是什么

提交不是“保存一个文件”，而是保存项目在某一时刻的快照。

一个 commit 里通常包含：

- 作者信息
- 提交信息
- 当前快照
- 父提交引用

所以 Git 历史本质上是：

- 一串有父子关系的快照链

---

## 8. 提交信息怎么写

好的提交信息应该做到：

- 说明这次改动的目的
- 让别人能快速理解
- 保持提交粒度单一

常见前缀：

- `feat`
- `fix`
- `docs`
- `refactor`
- `test`
- `chore`

例如：

```bash
git commit -m "fix: prevent duplicate submit on checkout form"
```

比写成：

```bash
git commit -m "update"
```

有用得多。

---

## 9. 小步提交的价值

小步提交有很多实际收益：

- review 更轻松
- 回滚更安全
- bisect 更有效
- 问题定位更快

所以一个好习惯是：

- 每次提交只做一件相对明确的事

这比“改了一整天最后一次性提交一大坨”更利于协作。

---

## 10. 查看历史与差异

常用命令：

```bash
git log
git log --oneline --graph --decorate
git diff
git diff --staged
```

它们分别常用于：

- 看历史
- 看提交关系
- 看工作区改动
- 看暂存区改动

习惯在提交前先看 `diff`，是非常重要的自检动作。

---

## 11. HEAD 是什么

`HEAD` 表示当前所在位置。

通常情况下：

- `HEAD` 指向当前分支
- 当前分支再指向最新提交

如果进入 detached HEAD 状态，意味着：

- 你当前直接指向某个提交
- 不是在一个正常分支上继续推进

这在查看历史或临时调试时很常见，但不适合长期开发。

---

## 12. 分支是什么

分支本质上只是：

- 指向某个提交的可移动指针

这个概念很重要，因为它说明：

- 分支并不神秘
- 创建分支通常很轻量

分支常见用途：

- 功能开发
- Bug 修复
- 版本发布
- 实验性改动

---

## 13. 常见分支操作

```bash
git branch
git switch main
git switch -c feature/login
```

常见理解：

- `branch`：看分支
- `switch`：切换分支
- `switch -c`：创建并切换新分支

现代 Git 中，用 `switch` 往往比老的 `checkout` 语义更清晰。

---

## 14. 远程仓库是什么

远程仓库是团队共享协作的历史存储点。

常见命令：

```bash
git remote -v
git fetch
git pull
git push
```

常见远程名：

- `origin`

远程仓库不是“另一个 Git 模式”，只是：

- 你和别人共享历史的地方

---

## 15. fetch、pull、push 的区别

### `git fetch`

只把远程历史抓下来，不自动合并到当前分支。

### `git pull`

通常等于：

- 先 `fetch`
- 再合并或 rebase

### `git push`

把本地分支提交推送到远程。

一个常见好习惯是：

- 不要一上来就 `pull`
- 先 `fetch` 看清楚变化，再决定怎么整合

---

## 16. 远程跟踪分支

比如：

- `origin/main`
- `origin/dev`

它们表示的是：

- 远程分支在本地的映射引用

这有助于你理解：

- 本地分支和远程分支不是同一个对象
- `fetch` 更新的是远程跟踪分支

---

## 17. merge 是什么

`merge` 的作用是把两个分支历史合并起来。

示例：

```bash
git switch main
git merge feature/login
```

它的特点：

- 保留真实分叉历史
- 不改写已有提交
- 历史结构更完整

对团队协作来说，merge 的优势是：

- 相对安全
- 不会重写共享历史

---

## 18. rebase 是什么

`rebase` 的核心作用是：

- 把当前分支的提交“接”到新的基底上

示例：

```bash
git switch feature/login
git rebase main
```

它会让历史看起来更线性，但本质上是：

- 重写提交历史

所以它很强，也很需要边界感。

---

## 19. merge 与 rebase 的区别

### merge

- 保留分叉历史
- 不改写历史
- 团队协作更稳妥

### rebase

- 历史更线性
- 会改写提交
- 整理个人分支历史很好用

经验原则：

- 私有分支可以放心 rebase
- 已共享分支慎用 rebase 后强推

---

## 20. 为什么说 rebase 会改写历史

因为 rebase 不是把旧提交“搬过去”，而是：

- 基于新的父提交重新生成一批提交

虽然改动内容可能一样，但提交 ID 会变。

这就是为什么：

- 别人已经基于这段历史工作时，你再改写它会很危险

---

## 21. 冲突是什么

冲突本质上是 Git 无法自动判断：

- 同一位置最终该保留哪个版本

常见冲突场景：

- merge
- rebase
- cherry-pick

冲突不是 Git “坏了”，而是提示你：

- 这里需要人工做业务判断

---

## 22. 冲突处理流程

常见流程：

1. 查看冲突文件
2. 理解两边改动意图
3. 手动保留正确结果
4. `git add` 标记已解决
5. 继续 merge / rebase / cherry-pick

关键不是“把标记删掉就行”，而是：

- 真正理解两边逻辑如何合并才对

冲突解决后一定要重新验证功能。

---

## 23. restore、reset、revert 的区别

这三个命令最容易混。

### `restore`

用于恢复文件内容或取消暂存。

### `reset`

移动分支指针，也可能影响暂存区和工作区。

### `revert`

创建一个反向提交来撤销已有提交。

共享历史里通常更推荐：

- `revert`

因为它不会改写已共享历史。

---

## 24. reset 的三种常见模式

### `--soft`

- 移动提交指针
- 保留暂存区和工作区

### `--mixed`

- 移动提交指针
- 重置暂存区
- 保留工作区

### `--hard`

- 同时影响提交、暂存区、工作区

最危险的是：

- `reset --hard`

它会直接丢弃未保留的工作区改动，使用前必须非常确定。

---

## 25. amend 是什么

```bash
git commit --amend
```

它通常用于：

- 修改刚刚那次提交的信息
- 把遗漏的小改动补进上一条提交

它的本质依然是：

- 重写最近一次提交

所以如果那次提交已经 push 并且被别人基于它工作，就要谨慎。

---

## 26. stash 是什么

`stash` 用于临时保存当前未完成改动。

常见命令：

```bash
git stash
git stash list
git stash pop
```

适合场景：

- 做到一半要切分支
- 先处理线上紧急问题
- 当前改动还不适合 commit

但不要把 stash 当长期储物箱，否则很容易忘。

---

## 27. cherry-pick 是什么

`cherry-pick` 用于把某个提交摘到当前分支。

```bash
git cherry-pick abc1234
```

适合场景：

- 只想把某个修复同步到发布分支
- 不想整个 merge 一个功能分支

它的价值在于：

- 精准搬运某个提交

但也要注意：

- 可能产生冲突
- 历史上会生成新的提交，不是复用原提交对象

---

## 28. tag 是什么

Tag 通常用于给某个提交打版本标记。

```bash
git tag v1.0.0
git push origin v1.0.0
```

常见用途：

- 发布版本
- 标记上线点
- 回滚对照

Tag 很适合作为发布体系的一部分。

---

## 29. `.gitignore`

`.gitignore` 用来忽略不应进入版本控制的文件。

常见内容：

- `node_modules/`
- `dist/`
- 日志文件
- 本地环境文件
- IDE 临时文件

要注意：

- `.gitignore` 只对“尚未被追踪的文件”生效
- 已经被 Git 跟踪的文件，不会因为后来加到 ignore 就自动消失

---

## 30. reflog 为什么很重要

`reflog` 是很多 Git 事故的救命工具。

```bash
git reflog
```

它记录的是：

- HEAD 和引用移动过的历史

适合找回这些情况：

- 误 reset
- 误删分支
- amend 后找旧提交
- rebase 后找回旧位置

很多“我提交没了”的场景，最后都能靠 reflog 找回来。

---

## 31. blame 与 bisect

### `git blame`

适合查看：

- 某一行代码最后是谁在什么提交改的

它有助于：

- 快速找到上下文和责任提交

### `git bisect`

适合：

- 在一段历史里二分定位哪个提交引入了 bug

这在回归问题排查时非常强大。

---

## 32. detached HEAD

当你直接 checkout 到某个提交时，可能进入 detached HEAD 状态。

这表示：

- 你当前不在某个正常分支上

临时查看历史没问题，但如果你在这里继续提交：

- 新提交可能变得不容易找回

如果需要继续工作，通常应该：

- 新建一个分支接住它

---

## 33. 分支协作模式

常见协作方式是 feature branch：

1. 从主分支切功能分支
2. 本地开发并提交
3. 同步主分支最新变更
4. 提交 PR / MR
5. review 后合并

这种方式的优点：

- 主分支更稳定
- review 更清晰
- 风险更可控

---

## 34. Pull Request / Merge Request 的意义

PR / MR 不只是“发起合并申请”，它还承载：

- 代码审查
- CI 校验
- 团队协作沟通
- 上下文留档

因此一个好的 PR 应该：

- 改动聚焦
- 描述清楚
- diff 可读
- 不混入无关改动

---

## 35. 为什么不要随意改写共享历史

共享历史一旦被别人拉走，你再改写它，就会导致：

- 别人的分支基于旧历史
- 再同步时出现大量冲突或重复提交

典型高风险操作：

- rebase 已共享分支
- amend 已共享提交
- force push 覆盖别人的工作

所以一定要问自己：

- 这段历史是不是只有我一个人在用

---

## 36. force push 什么时候能用

`push --force` 不是绝对不能用，但必须有边界。

更安全的做法通常是：

- `push --force-with-lease`

它能避免把别人刚推上去的提交直接覆盖掉。

适合场景：

- 你在整理自己的私有功能分支历史

不适合场景：

- 主分支
- 团队共享开发分支

---

## 37. 发布与回滚

Git 在发布流程里通常扮演这些角色：

- 标记版本
- 记录发布点
- 追踪热修复
- 支持快速回滚

常见做法：

- 用 tag 标记版本
- 用 release branch 管理发布
- 用 revert 回滚危险提交

理解 Git 后，发布和回滚都会更有把控感。

---

## 38. 常见问题排查思路

遇到 Git 问题时，可以先问自己：

1. 我当前在哪个分支
2. `HEAD` 指向哪里
3. 改动在工作区、暂存区还是已经提交
4. 这段历史有没有 push
5. 我是想恢复文件，还是撤销提交，还是找回历史

很多时候不是命令难，而是没有先定位自己当前所处状态。

---

## 39. 常见误区

- 把 `pull` 当万能同步命令
- 不看 diff 就直接 commit
- 一次提交塞太多事情
- 共享分支上随意 rebase / force push
- 误以为 `.gitignore` 能删除已追踪文件
- 遇到冲突只想“怎么消掉标记”，不理解业务含义

这些误区会直接影响协作质量。

---

## 40. Git 最佳实践

- 一个提交只做一件事
- 提交前先看 `git diff`
- 多用 feature branch
- 私有分支可整理历史，共享分支谨慎改写
- 冲突解决后认真验证
- 发布用 tag 标记
- 出事故先看 reflog，不要慌着乱操作

Git 真正的价值不只是记录历史，而是让团队能更稳地演进代码。

---

## 41. Git 最值得掌握的主线

如果只抓一条主线，就抓这条：

1. 明白改动当前在哪一层
2. 明白每个命令会影响哪一层
3. 明白哪些操作会改写历史
4. 明白共享历史和私有历史的边界
5. 出问题时能用 diff、log、reflog 把状态找回来

做到这些，Git 就不会再像一堆零散命令，而会变成你真正能驾驭的版本控制工具。
