# 部署与运维基础核心知识点

很多前端学习会停在“本地跑起来”，但真实项目能不能稳定上线、怎么配置环境、为什么缓存不生效、为什么 CDN 旧资源没刷新、为什么灰度发布后部分用户报错，这些都属于部署与运维基础。

所以这一章的核心不是让前端变成运维，而是让前端真正理解：

- 产物上线后会经历什么
- 静态资源如何被访问和缓存
- 域名、证书、反向代理、CDN、监控、回滚是怎么协同的

---

## 1. 前端部署到底在部署什么

前端部署通常不是“把源码传上去”，而是：

- 把构建产物发布到可访问环境

这些产物常见包括：

- `index.html`
- JS/CSS chunk
- 图片字体等静态资源
- Source Map

部署的目标是：

- 让用户稳定访问到正确版本

---

## 2. 静态资源部署的本质

大多数前端应用，最终都是以静态资源形式被分发。

即使是复杂 SPA，浏览器最终获取的通常也是：

- HTML
- JS
- CSS
- 图片

所以前端部署最核心的是：

- 静态资源分发
- 路由支持
- 缓存策略

---

## 3. 为什么构建产物和源码要分开理解

源码里有：

- TypeScript
- Vue / React 组件
- 模块化结构

上线时用户拿到的是：

- 已构建和压缩后的产物

因此部署关注的是：

- 产物目录
- 文件命名
- hash 缓存
- Source Map

而不是源码本身结构。

---

## 4. Nginx 在前端部署中的角色

Nginx 在前端场景里常见作用包括：

- 提供静态资源
- 做反向代理
- 配置缓存头
- 处理 gzip / brotli
- 处理 HTTPS
- 路由重写

所以前端即使不专职做运维，也很值得理解 Nginx 的基本角色。

---

## 5. 反向代理是什么

反向代理可以理解为：

- 用户请求先到代理服务器
- 代理再转发给真实服务

常见用途：

- 前后端分流
- 同域代理接口
- 隐藏真实后端服务地址
- 负载均衡

很多“前端请求 `/api` 就能通”的背后，其实是代理在工作。

---

## 6. SPA 路由部署为什么常出问题

单页应用使用 history 路由时，浏览器直接访问：

- `/users/1`

服务器如果按静态文件路径查找，可能会得到：

- 404

因此常见解决方式是：

- 对非真实静态资源请求统一回退到 `index.html`

这就是 SPA 路由部署中最经典的一类配置问题。

---

## 7. CDN 是什么

CDN 是内容分发网络。

它的作用通常包括：

- 就近分发静态资源
- 降低源站压力
- 提升加载速度
- 提供缓存能力

对于前端来说，CDN 往往是性能和部署体系的重要组成部分。

---

## 8. CDN 与缓存策略

CDN 的价值不只是“更快”，还在于：

- 可以缓存静态资源
- 降少回源请求

前端常见实践：

- 带 hash 的静态资源长缓存
- HTML 入口页短缓存或协商缓存

如果缓存策略不合理，就可能出现：

- 用户一直拿旧版本

---

## 9. 文件 hash 为什么重要

前端构建产物通常会带内容 hash，例如：

- `index.a1b2c3.js`

它的作用：

- 内容没变时文件名不变，可长期缓存
- 内容变化时文件名变化，避免旧缓存污染

这是现代前端部署里非常关键的版本控制策略。

---

## 10. HTML 和静态资源的缓存策略为什么不同

原因在于：

- HTML 是入口，通常需要尽快指向最新资源清单
- 带 hash 的 JS/CSS 静态资源可以放心长缓存

所以常见做法是：

- `index.html` 不长时间强缓存
- 静态资源长缓存

如果这两类资源都一刀切，很容易出问题。

---

## 11. HTTPS 和证书基础

线上站点基本都要 HTTPS。

HTTPS 的价值：

- 加密传输
- 防篡改
- 身份认证

这依赖：

- 证书
- TLS 握手

前端需要理解：

- 证书过期、域名不匹配、混合内容，都会直接影响页面访问

---

## 12. 混合内容问题

如果页面是 HTTPS，但里面加载了 HTTP 资源，就会出现混合内容问题。

例如：

- 脚本
- 图片
- 接口

浏览器通常会：

- 警告
- 甚至阻止加载

所以前端部署和资源引用必须保持协议一致。

---

## 13. 环境变量与运行时配置

部署时经常需要区分：

- 开发环境
- 测试环境
- 预发环境
- 生产环境

需要配置的可能包括：

- API 地址
- 埋点开关
- 功能开关

这里要区分：

- 构建时注入
- 运行时配置

这两者很多团队一开始容易混。

---

## 14. 为什么说前端可见配置不是秘密

所有进入浏览器的前端配置，本质上都是可见的。

因此：

- 不要把真正密钥放在前端环境变量里

前端配置适合放：

- 公共 API 地址
- 公共开关
- 公开埋点配置

不适合放：

- 真正敏感凭证

---

## 15. 制品是什么

制品可以理解为：

- 构建完成后可用于部署的稳定产物

例如：

- 前端打包后的静态资源包
- Docker 镜像

制品思维很重要，因为它代表：

- 从“构建”到“发布”的中间边界

---

## 16. Source Map 上报与线上调试

线上错误定位通常离不开 Source Map。

但更合理的方式一般是：

- 上传到错误平台
- 不直接公网暴露

这样可以兼顾：

- 调试效率
- 源码安全

---

## 17. 前端监控为什么属于部署后能力

部署不是结束，上线后还需要持续观察。

常见监控包括：

- JS 错误
- 资源加载错误
- 接口异常
- 页面性能指标
- 白屏或崩溃问题

没有监控，很多线上问题只能靠用户反馈，成本很高。

---

## 18. 日志、告警与排障闭环

真正成熟的线上体系通常还会有：

- 错误日志
- 性能日志
- 告警规则
- 版本追踪

这样发生问题时，团队才能快速知道：

- 哪个版本出问题
- 影响范围多大
- 是否需要回滚

---

## 19. 灰度发布是什么

灰度发布指的是：

- 新版本先只给部分用户使用

它的价值：

- 小范围验证
- 控制风险
- 降低全量事故概率

对前端来说，灰度常和这些能力结合：

- 路由分流
- 用户分桶
- 特性开关

---

## 20. 回滚为什么要提前设计

上线出问题后最怕的是：

- 没有明确回滚方案

常见回滚方式：

- 回滚静态资源版本
- 切回旧制品
- 关闭特性开关
- 切回旧 CDN 清单

真正好的部署体系不是“不会出错”，而是：

- 出错时能快速恢复

---

## 21. 蓝绿发布与滚动发布

这些概念更多见于整体发布体系，但前端也值得理解。

### 蓝绿发布

新旧环境并行，切流量切换版本。

### 滚动发布

逐步替换实例。

理解这些方式，有助于你看懂：

- 为什么某些用户先看到新版本
- 为什么回滚有时能更快

---

## 22. 前端资源 404 的常见原因

线上资源 404 常见于：

- 发布目录错误
- 公共路径配置错误
- CDN 未同步
- HTML 指向了旧资源清单
- hash 资源被删但 HTML 仍引用

这类问题的排查，通常需要结合：

- 构建配置
- 部署目录
- CDN 缓存

---

## 23. 缓存失效与强制刷新

前端发布最常见的问题之一就是：

- 用户拿到旧缓存

常见处理方式：

- 静态资源文件名带 hash
- 控制 HTML 缓存
- 必要时 CDN 刷新

不要把“让用户手动 Ctrl+F5”当成正式方案。

---

## 24. 前端与后端联动部署

真实系统里，前端上线常常不是单独发生的。

例如：

- 接口字段变化
- 权限逻辑变化
- 新旧版本兼容窗口

因此部署时还要考虑：

- 前后端版本兼容策略
- 接口灰度
- 回滚是否双向协调

---

## 25. Docker 与前端部署的关系

前端项目并不一定必须 Docker 化，但很多团队会使用 Docker 来统一：

- 构建环境
- 运行环境
- 发布制品

对前端来说，更重要的是理解：

- 它是在帮你稳定环境和交付流程

而不是把 Docker 当成“另一个构建工具”。

---

## 26. 运维协同为什么重要

很多线上问题并不是纯前端问题，也不是纯运维问题。

例如：

- 缓存不生效
- 证书异常
- 代理转发错误
- 域名配置错误

这说明前端工程师很有必要理解基础运维语言，这样才能更高效协同。

---

## 27. 部署与运维常见误区

- 以为构建成功就等于上线没问题
- 以为 CDN 只负责加速
- 以为 HTTPS 配好一次就结束
- 以为前端不需要懂 Nginx
- 以为 Source Map 可以直接公开
- 以为出问题时总能临时手工修

这些误区在真实线上环境里代价很高。

---

## 28. 部署与运维基础最值得掌握的主线

如果只抓一条主线，就抓这条：

1. 明白前端上线的是构建产物而不是源码
2. 明白静态资源、路由、缓存、CDN 如何协同工作
3. 明白 HTTPS、代理、证书、环境配置如何影响访问
4. 明白监控、灰度、回滚为什么是发布体系的一部分
5. 明白前端上线后仍然需要持续观察和治理

把这条主线建立起来，你就不会再把部署理解成“丢文件上服务器”，而会真正看懂前端系统如何在线上稳定运行。
