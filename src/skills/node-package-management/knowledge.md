# Node.js 与包管理核心知识点

前端工程几乎离不开 Node.js 和包管理。即使你主要写浏览器代码，项目安装依赖、运行脚本、构建打包、测试执行、代码规范、发布包，背后基本都依赖 Node 生态。

所以学这一章，关键不是背 `npm install`，而是理解：

- Node.js 到底是什么
- 前端为什么离不开它
- 模块系统为什么总在 CommonJS 和 ESM 之间切换
- 包管理器到底帮我们管理了什么
- 为什么 lockfile、semver、monorepo、workspace 会影响团队协作

---

## 1. Node.js 是什么

Node.js 是一个让 JavaScript 可以运行在浏览器之外的运行时环境。

它的价值在于：

- 让 JS 可以做服务端、脚本、构建工具、CLI

前端日常最常接触的其实是 Node 的“工程化角色”，例如：

- 安装依赖
- 跑 Vite / Webpack
- 跑测试
- 跑脚本
- 构建项目

---

## 2. 为什么前端要学 Node.js

因为现代前端项目并不只是浏览器代码。

真实项目里经常会碰到：

- 写构建脚本
- 调整 npm scripts
- 读环境变量
- 处理文件系统
- 调 CI 里的 Node 命令
- 发布组件库

不会 Node，很多前端工程问题就只能停留在“会用模板”，很难真正排查和扩展。

---

## 3. Node.js 与浏览器环境的区别

虽然都能运行 JavaScript，但运行环境不同。

浏览器更关注：

- DOM
- BOM
- 页面渲染

Node 更关注：

- 文件系统
- 进程
- 网络服务
- 命令行

例如浏览器有：

- `window`
- `document`

Node 则有：

- `process`
- `fs`
- `path`

所以同样是 JS，能用的 API 并不一样。

---

## 4. Node.js 常见内置能力

Node 常见内置模块包括：

- `fs`
- `path`
- `http`
- `https`
- `os`
- `process`
- `events`
- `stream`

前端工程里最常碰到的是：

- 路径处理
- 文件读写
- 环境变量
- 命令行参数

---

## 5. `process` 和环境变量

`process` 是 Node 中非常重要的全局对象。

常见用途：

- 获取运行参数
- 获取环境变量
- 获取当前工作目录

例如：

```js
console.log(process.cwd())
console.log(process.env.NODE_ENV)
```

前端工程化中，很多脚本行为都依赖环境变量。

---

## 6. Node.js 中的模块系统

Node 里最重要的一个话题就是模块系统。

常见两套：

- CommonJS
- ES Modules

这也是为什么很多项目里总会遇到：

- `require`
- `module.exports`
- `import`
- `export`

---

## 7. CommonJS

CommonJS 是 Node 早期最常见的模块系统。

写法：

```js
const utils = require('./utils')
module.exports = {}
```

特点：

- 更偏运行时加载
- 长期服务于 Node 生态

很多老项目和很多 npm 包依然大量使用它。

---

## 8. ES Modules

ESM 是现代 JavaScript 标准模块系统。

写法：

```js
import { sum } from './utils'
export default App
```

特点：

- 静态结构更强
- 更利于 Tree Shaking
- 浏览器原生支持

现代前端构建工具和现代包生态越来越偏向 ESM。

---

## 9. CommonJS 和 ESM 的常见差异

需要重点理解的点包括：

- 导入导出语法不同
- 加载机制不同
- 顶层 `this` 行为不同
- 默认导出和互操作处理更复杂

真实工程里最大的问题往往不是“背区别”，而是：

- 混用时兼容性容易出问题

---

## 10. `package.json` 的意义

`package.json` 是 Node 和前端项目的核心清单文件。

常见作用：

- 定义项目名称和版本
- 记录依赖
- 定义 scripts
- 配置模块入口
- 配置包发布信息

可以把它理解为：

- 项目的依赖和运行元信息中心

---

## 11. `dependencies`、`devDependencies`、`peerDependencies`

这是包管理里非常高频的概念。

### `dependencies`

运行项目本身需要的依赖。

### `devDependencies`

开发和构建阶段需要的依赖。

### `peerDependencies`

表示当前包依赖宿主环境提供某个依赖版本。

例如组件库常声明：

- React 作为 peer dependency

因为它不希望自己打包一份 React。

---

## 12. `scripts` 是什么

`scripts` 用于定义项目命令入口。

例如：

```json
{
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "test": "vitest"
  }
}
```

它的价值是：

- 把团队常用命令统一起来
- 让复杂命令可复用
- 让 CI 更容易接入

---

## 13. npm、yarn、pnpm 是什么

它们都是包管理器。

主要职责：

- 安装依赖
- 解析依赖树
- 写 lockfile
- 执行 scripts

它们的核心差异通常体现在：

- 安装速度
- 依赖存储方式
- workspace 能力
- 磁盘占用

---

## 14. npm、yarn、pnpm 的常见区别

### npm

- 官方默认
- 生态最普适

### yarn

- 曾长期在 workspace 和速度方面有优势

### pnpm

- 更强调磁盘复用和依赖隔离
- 现在在很多前端项目中非常流行

团队里最重要的不是“谁绝对更强”，而是：

- 统一工具和规范

---

## 15. lockfile 为什么重要

lockfile 用来锁定依赖安装结果。

常见文件：

- `package-lock.json`
- `yarn.lock`
- `pnpm-lock.yaml`

它的价值是：

- 保证团队和 CI 安装出的依赖版本尽量一致
- 减少“我本地能跑你那边不能跑”的问题

所以 lockfile 一般应该进入版本控制。

---

## 16. semver 是什么

Semver 是语义化版本规范。

格式通常是：

- `major.minor.patch`

例如：

- `1.2.3`

含义通常是：

- major：不兼容变更
- minor：向后兼容的新功能
- patch：向后兼容的修复

这是依赖管理的基础语言。

---

## 17. `^`、`~`、固定版本的区别

依赖版本常见写法：

- `^1.2.3`
- `~1.2.3`
- `1.2.3`

大致理解：

- `^`：允许次版本和补丁版本更新
- `~`：通常只允许补丁版本更新
- 固定版本：只装指定版本

如果团队对稳定性要求高，依赖升级就更需要明确策略，而不能完全放任漂移。

---

## 18. 依赖树为什么复杂

前端项目装一个包，往往会带出很多间接依赖。

因此项目依赖实际上是：

- 直接依赖 + 传递依赖

问题也常出在这里，例如：

- 版本冲突
- 重复依赖
- 漏洞出现在传递依赖里

所以“我没装这个包但项目里为什么有”是很常见的现象。

---

## 19. 去重与重复依赖

重复依赖会带来：

- 磁盘占用增加
- 安装变慢
- bundle 体积变大
- 运行时实例不一致

特别是在 React、Vue、状态库这类依赖上，重复实例可能直接造成问题。

这也是为什么：

- peerDependencies
- 包去重
- workspace 管理

都非常重要。

---

## 20. `node_modules` 到底是什么

`node_modules` 是依赖安装后的本地目录。

它不是简单“缓存区”，而是：

- 当前项目运行时可解析到的依赖空间

不同包管理器组织它的方式不同，但它共同承载：

- 模块解析
- 依赖层级
- 可执行命令链接

---

## 21. 为什么不建议手改 `node_modules`

因为：

- 依赖是安装产物，不是源码主仓库
- 一次重新安装就会被覆盖
- 无法稳定同步给团队

如果确实需要定制第三方包，通常应该考虑：

- patch 方案
- fork
- 等官方修复

而不是直接改安装结果。

---

## 22. `npx` 和包执行

`npx` 常用于：

- 临时执行本地或远程包提供的命令

例如：

```bash
npx eslint .
```

它减少了：

- 全局安装依赖的需求

现代前端开发里，很多 CLI 都通过这种方式执行。

---

## 23. 全局安装和本地安装的区别

### 全局安装

适合：

- 通用 CLI
- 和具体项目无关的工具

### 本地安装

适合：

- 项目依赖工具链
- 团队需要一致版本的命令

工程上通常更推荐：

- 项目依赖尽量本地安装

这样更容易保证一致性。

---

## 24. workspace 是什么

Workspace 用于在一个仓库中管理多个包。

适合场景：

- Monorepo
- 组件库 + demo
- 多个应用共享包

它的价值：

- 统一依赖安装
- 本地包联动开发
- 降低跨仓库同步成本

---

## 25. Monorepo 和多仓库的区别

### Monorepo

多个包或应用放在一个仓库管理。

优点：

- 共享代码更方便
- 统一依赖和规范
- 跨包改动更直接

### 多仓库

每个包独立仓库。

优点：

- 边界清晰
- 独立发布更自然

没有绝对标准，关键看团队规模和协作模式。

---

## 26. 包发布的核心字段

如果你要发布 npm 包，常见需要关注：

- `name`
- `version`
- `main`
- `module`
- `types`
- `exports`
- `files`

这些字段会影响：

- 包如何被导入
- 类型文件是否可见
- 发布产物是否正确

---

## 27. `main`、`module`、`exports` 的作用

### `main`

传统入口字段，通常指向 CommonJS 入口。

### `module`

很多工具链约定的 ESM 入口字段。

### `exports`

更现代、更严格的导出控制方式。

它可以决定：

- 哪些路径允许外部导入
- 针对不同环境暴露什么入口

现在发布现代包时，`exports` 越来越重要。

---

## 28. 类型声明与 `types`

TypeScript 包通常需要提供类型声明。

常见方式：

- 自带 `.d.ts`
- 通过 `types` 指向声明入口

如果类型文件缺失，使用方体验会明显变差。

所以发布前端包时，类型声明几乎是基础要求。

---

## 29. CLI 包是怎么工作的

很多 npm 包不仅能被 import，也能作为命令行工具使用。

它们通常通过：

- `bin`

字段暴露可执行命令。

例如一个 CLI 包安装后，就可能在 `npm run` 或终端中直接执行对应命令。

这也是为什么很多前端工具看起来像“命令”，本质上其实是 Node 包。

---

## 30. 缓存、镜像与安装速度

包安装速度受很多因素影响：

- 网络环境
- registry 源
- lockfile 稳定性
- 包管理器缓存
- 依赖体量

常见做法：

- 使用合适镜像源
- 用 CI 缓存依赖目录
- 控制依赖数量

安装慢有时不是工具本身慢，而是依赖体系过重。

---

## 31. 依赖漏洞与安全扫描

Node 生态里，供应链安全很重要。

常见问题：

- 依赖存在已知漏洞
- 传递依赖带来风险
- 恶意包投毒

常见治理方式：

- 审查依赖来源
- 漏洞扫描
- 及时升级
- 锁定版本
- 对关键依赖建立准入机制

---

## 32. 为什么前端项目也要关心 Node 版本

不同 Node 版本会影响：

- ESM 行为
- 内置 API 可用性
- 脚本运行结果
- 构建工具兼容性

因此团队通常需要：

- 统一 Node 版本

常见方式：

- `.nvmrc`
- `engines`
- CI 固定版本

---

## 33. 前端项目里的包管理最佳实践

- 统一包管理器
- lockfile 提交到仓库
- 统一 Node 版本
- 谨慎升级大依赖
- 减少无意义依赖
- 本地工具尽量项目内安装
- 重要包明确 peerDependencies 设计

这些事情虽然不炫，但对团队稳定性非常关键。

---

## 34. 常见误区

- 以为 Node.js 就等于后端开发
- 以为 `node_modules` 只是缓存
- 以为 lockfile 不重要
- 以为依赖升级只是“顺手更新”
- 以为 `peerDependencies` 和 `dependencies` 一样
- 以为项目能跑就说明依赖体系没问题

这些误区会直接影响工程稳定性。

---

## 35. Node.js 与包管理最值得掌握的主线

如果只抓一条主线，就抓这条：

1. Node 提供了什么运行时能力
2. 包管理器如何安装和解析依赖
3. 模块系统如何影响构建和运行
4. lockfile、semver、workspace 如何影响团队协作
5. 如何安全、稳定地管理依赖和发布包

把这条主线建立起来，Node.js 与包管理就不再只是“装依赖的黑盒”，而会成为你真正能掌控的工程基础。
