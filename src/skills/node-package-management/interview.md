# Node.js 与包管理面试题

## Q1. Node.js 是什么？前端为什么要学它？

**答：**

Node.js 是一个让 JavaScript 运行在浏览器之外的运行时环境。

前端之所以要学它，是因为现代前端工程大量依赖 Node 生态，例如：

- 安装依赖
- 跑构建工具
- 执行测试
- 写脚本
- 发布包

所以即使主要做浏览器开发，Node 也是前端工程基础设施的一部分。

---

## Q2. CommonJS 和 ES Modules 的区别是什么？

**答：**

### CommonJS

常见写法：

- `require`
- `module.exports`

更偏 Node 早期生态，运行时加载。

### ES Modules

常见写法：

- `import`
- `export`

是现代 JavaScript 标准模块系统，静态结构更强，也更利于 Tree Shaking。

实际工程里最大的难点往往不是定义，而是：

- 混用时兼容性问题

---

## Q3. `dependencies`、`devDependencies`、`peerDependencies` 有什么区别？

**答：**

- `dependencies`：运行时需要
- `devDependencies`：开发和构建时需要
- `peerDependencies`：希望宿主环境提供的依赖

比如组件库常把 React 声明成 `peerDependencies`，因为它不希望自己再打包一份 React。

---

## Q4. lockfile 为什么重要？

**答：**

lockfile 用来锁定依赖解析结果，保证团队和 CI 安装出的依赖版本尽量一致。

它的重要价值是：

- 降低环境不一致
- 减少随机升级带来的问题
- 提高构建可复现性

所以 lockfile 一般应该提交到仓库。

---

## Q5. `^1.2.3`、`~1.2.3`、固定版本有什么区别？

**答：**

- `^1.2.3`：通常允许次版本和补丁版本升级
- `~1.2.3`：通常只允许补丁版本升级
- `1.2.3`：固定到这个版本

这背后体现的是依赖升级策略，不同团队对稳定性和更新速度的偏好不同。

---

## Q6. npm、yarn、pnpm 的区别怎么答？

**答：**

它们都是包管理器，核心职责类似：

- 安装依赖
- 管理 lockfile
- 执行 scripts

区别主要在于：

- 安装性能
- 存储方式
- workspace 能力
- 依赖隔离策略

比如 pnpm 更强调磁盘复用和依赖隔离，因此现在很多前端项目会优先选择它。

---

## Q7. `peerDependencies` 为什么容易出问题？

**答：**

因为它不是真正由当前包自己安装并封装好的依赖，而是要求宿主环境提供。

如果宿主环境：

- 没装
- 版本不匹配

就可能出现：

- 安装警告
- 运行时异常
- 多实例问题

所以设计库时，`peerDependencies` 需要非常谨慎。

---

## Q8. Monorepo 和 workspace 有什么关系？

**答：**

Monorepo 是多个项目或包放在同一个仓库里的组织方式。

Workspace 是包管理器提供的一种能力，用来更方便地管理一个仓库中的多个包。

可以理解为：

- Monorepo 是仓库组织形式
- Workspace 是实现这类管理的重要工具能力

---

## Q9. `main`、`module`、`exports` 分别是干什么的？

**答：**

- `main`：传统入口，常用于 CommonJS
- `module`：很多工具链约定的 ESM 入口
- `exports`：更现代的导出控制字段，可精细控制对外可见入口

现在越来越推荐理解并使用 `exports`，因为它更规范也更可控。

---

## Q10. 为什么前端项目也要统一 Node 版本？

**答：**

因为不同 Node 版本会影响：

- 模块系统行为
- 内置 API 可用性
- 构建工具兼容性
- 脚本执行结果

如果团队和 CI 的 Node 版本不一致，就容易出现：

- 你本地能跑，CI 过不了

所以统一 Node 版本是很重要的工程实践。
