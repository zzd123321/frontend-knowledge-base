# Vue 面试题


## Q1. Vue 的核心特性是什么？

**答：**

Vue 的核心特性可以概括为三点：声明式渲染、组件化开发、响应式数据驱动视图。

声明式渲染意味着开发者不用频繁手动操作 DOM，而是描述“当前状态下界面应该长什么样”，框架负责把状态和界面同步起来。组件化则意味着我们把页面拆成可复用、可组合、边界清晰的组件单元，降低复杂度。响应式系统则是 Vue 最底层的重要能力，它负责追踪数据变化，并把变化自动同步到界面和依赖逻辑中。

如果面试官追问一句话总结，我通常会说：Vue 是一个基于响应式系统、以组件化和声明式渲染为核心的渐进式前端框架。

## Q2. Vue 2 和 Vue 3 的主要区别是什么？

**答：**

这题建议不要只说“Vue 3 更快”，而是从几个维度回答。

### 1. 响应式实现不同

- Vue 2 主要基于 `Object.defineProperty`
- Vue 3 主要基于 `Proxy`

Vue 3 因为有 `Proxy`，在处理对象新增属性、删除属性、数组和集合类型时更加自然，也更容易做能力扩展。

### 2. API 设计不同

- Vue 2 以 Options API 为主
- Vue 3 同时支持 Options API 和 Composition API

Composition API 让逻辑复用和复杂逻辑组织更加清晰，也更适合 TypeScript。

### 3. 性能和包体

Vue 3 在编译、运行时和 Tree Shaking 上都做了大量优化，因此在现代工程体系下通常更有优势。

### 4. 工程化生态

Vue 3 和 Vite、Pinia、`<script setup>`、TS 配合更自然，已经成为现代 Vue 项目的主流组合。

所以更完整的结论是：Vue 3 不是只做了性能升级，而是对响应式底层、API 组织方式和现代工程化体验做了整体升级。

## Q3. 什么是 MVVM？Vue 如何体现这一思想？

**答：**

MVVM 指的是：

- Model：数据模型
- View：视图
- ViewModel：连接视图与数据的中间层

在 Vue 中，开发者定义的数据状态，比如 `data`、`ref`、`reactive`，可以看作 Model；模板 `template` 是 View；Vue 的响应式系统、依赖追踪、模板编译和组件实例可以理解为 ViewModel。

Vue 如何体现 MVVM？最典型的地方在于双向同步思想：

- 数据变化，视图自动更新
- 用户操作触发输入变化，数据也能同步更新

但更严谨地说，Vue 不是“完全面向双向绑定”的框架，它本质上仍强调单向数据流和响应式驱动，只是在表单场景下通过 `v-model` 提供了方便的双向绑定语法糖。

## Q4. Vue 的响应式原理是什么？

**答：**

Vue 3 的响应式核心可以简化理解为三步：

1. 通过 `Proxy` 拦截对象的读取和设置
2. 在读取时收集依赖
3. 在设置时触发依赖更新

更细一点说，当某个组件渲染过程中读取了响应式数据，Vue 会记录“这个组件依赖了这个数据”。当这个数据被修改时，Vue 就能找到依赖它的副作用函数或组件更新函数，安排它重新执行。

Vue 2 的原理类似，但底层是用 `Object.defineProperty` 劫持对象属性，因此在：

- 新增属性
- 删除属性
- 某些数组操作

这些场景上不如 Vue 3 自然。

面试里如果再往下问，你还可以补一句：Vue 3 的响应式系统不仅能驱动模板更新，也能驱动 `computed`、`watchEffect` 等副作用机制。

## Q5. `ref` 和 `reactive` 有什么区别？

**答：**

这题不要只回答“一个有 `.value`，一个没有”，那样太浅。

### `ref`

`ref` 可以包裹基础类型，也可以包裹对象。它返回一个带 `.value` 的响应式包装对象。

```ts
const count = ref(0)
```

优势是表达统一，尤其在组合式函数中更稳定，也更适合整体替换值。

### `reactive`

`reactive` 更适合对象、数组这类引用类型，它直接返回代理后的对象。

```ts
const user = reactive({ name: 'Tom' })
```

它使用起来更像“直接操作对象”，但也有典型陷阱，比如直接解构容易丢失响应式。

### 实战建议

我的经验是：

- 基础类型优先 `ref`
- 对象结构可用 `reactive`
- 如果团队追求一致性，也可以大部分场景统一偏向 `ref`

真正关键不是死记硬背谁更高级，而是知道各自的行为特征和坑点。

## Q6. `computed` 和 `watch` 有什么区别？

**答：**

这题是 Vue 高频核心题。

### `computed`

`computed` 用于根据已有响应式状态推导出一个派生值。它的特点是：

- 有缓存
- 依赖不变时不会重复计算
- 更适合表达“状态的派生结果”

```ts
const fullName = computed(() => first.value + last.value)
```

### `watch`

`watch` 用于观察数据变化并在变化后执行副作用。它更适合：

- 请求接口
- 同步本地存储
- 响应路由变化
- 处理异步逻辑

### 本质区别

一句话总结：

- `computed` 解决“我要得到一个新值”
- `watch` 解决“值变了以后我要做一件事”

如果一个场景只是派生结果，却写成 `watch + setState`，那通常说明设计不够优雅。

## Q7. `watch` 和 `watchEffect` 的区别是什么？

**答：**

二者都能做副作用，但使用心智不一样。

### `watch`

- 依赖源显式指定
- 能拿到新值和旧值
- 可控性更强
- 更适合精确监听某些状态

### `watchEffect`

- 不需要手动写依赖源
- 会自动收集回调中用到的响应式依赖
- 写法更简洁
- 但依赖关系更隐式

如果逻辑比较简单、依赖也很明确，`watchEffect` 很顺手；如果你需要明确监听谁、拿新旧值、控制触发时机，`watch` 更适合。

面试中成熟的回答通常还会加一句：项目里如果副作用逻辑较复杂，我更倾向优先用 `watch`，因为可读性和可维护性更好。

## Q8. `v-if` 和 `v-show` 的区别是什么？

**答：**

### `v-if`

`v-if` 是真正的条件渲染。条件为假时，节点不会被创建；条件变真时，才会创建。

优点：

- 初始不渲染时更节省
- 适合低频切换

缺点：

- 切换要创建和销毁节点，成本更高

### `v-show`

`v-show` 本质是切换元素的 `display`。

优点：

- 节点一直存在，切换开销小
- 适合高频显示隐藏

缺点：

- 初始就会渲染

所以实战判断原则是：

- 低频切换、可能一开始就不需要渲染：`v-if`
- 高频切换：`v-show`

## Q9. 为什么 `v-for` 需要 `key`？为什么不推荐用 index？

**答：**

`key` 的核心作用是标识节点身份，帮助 Vue 在 Diff 过程中更准确地复用、移动和销毁节点。

如果没有稳定的 `key`，框架只能尽量“就地复用”，这在静态列表可能没问题，但在有插入、删除、排序的动态列表中容易出现：

- 输入框内容串位
- 子组件内部状态错乱
- 动画表现异常

为什么不推荐 `index`？

因为 `index` 不是稳定身份，它会随着数组变化而变化。  
一旦中间插入或删除元素，后面所有项的 index 都会变，框架就可能把“原本属于 A 的 DOM 或组件状态”复用给 B。

成熟实践就是：尽量使用业务唯一 id 做 `key`。

## Q10. Vue 生命周期有哪些？常用的是哪些？

**答：**

Vue 3 组合式 API 常见生命周期包括：

- `onBeforeMount`
- `onMounted`
- `onBeforeUpdate`
- `onUpdated`
- `onBeforeUnmount`
- `onUnmounted`
- `onActivated`
- `onDeactivated`
- `onErrorCaptured`

### 最常用的是哪些？

最常用的一般是：

- `onMounted`
- `onUnmounted`

`onMounted` 适合：

- 请求首屏数据
- 访问真实 DOM
- 初始化第三方实例

`onUnmounted` 适合：

- 清理事件监听
- 清理定时器
- 断开订阅
- 销毁实例

这题真正考察的是你有没有“副作用生命周期管理”意识，而不是背 API 名字。

## Q11. Vue 组件通信有哪些方式？

**答：**

这是典型体系题，建议按层级回答。

### 1. 父传子：props

最基础也最常用，适合单向传值。

### 2. 子传父：emit

子组件通过事件通知父组件。

### 3. 双向绑定：`v-model`

适合输入组件、弹窗显隐等场景。

### 4. 跨层级：`provide` / `inject`

适合多层嵌套中的上下文共享。

### 5. 全局共享：Pinia

适合跨页面、跨模块的共享状态。

### 6. 模板 ref + `defineExpose`

适合父组件主动调用子组件能力，如弹窗的 `open()`。

如果面试官追问“你最推荐哪种”，一般回答：

- 父子通信优先 `props + emit`
- 跨层级上下文用 `provide/inject`
- 全局共享状态用 Pinia

## Q12. `v-model` 的原理是什么？

**答：**

`v-model` 本质是语法糖。

在原生表单元素上，它通常可以理解为：

- 把数据绑定到 `value`
- 监听输入事件再把值更新回去

在 Vue 3 自定义组件上，默认约定是：

- prop：`modelValue`
- event：`update:modelValue`

也就是说：

```vue
<Child v-model="visible" />
```

本质上近似于：

```vue
<Child
  :modelValue="visible"
  @update:modelValue="visible = $event"
/>
```

这题如果答到“本质是 prop + emit 的语法糖”，一般就到位了；如果再补充 Vue 3 允许多个 `v-model`，会更完整。

## Q13. 什么是插槽？具名插槽和作用域插槽是什么？

**答：**

插槽可以理解成：子组件把部分渲染位置交给父组件决定。

### 默认插槽

最基础，用于分发一块默认内容区域。

### 具名插槽

当一个组件有多个内容区域时，可以通过名字区分，比如 `header`、`footer`。

### 作用域插槽

作用域插槽的本质是：子组件把内部数据暴露给父组件，父组件再决定怎么渲染。

这在列表组件、表格组件、下拉组件、表单项组件里很常见。

一句话总结：

- 默认插槽：传内容
- 具名插槽：传到指定位置
- 作用域插槽：子组件提供数据，父组件提供渲染方式

## Q14. Composition API 和 Options API 有什么区别？

**答：**

这题建议从“代码组织方式”和“逻辑复用方式”来答。

### Options API

按配置项分类组织代码：

- `data`
- `methods`
- `computed`
- `watch`

优点：

- 上手直观
- 小组件阅读成本低

缺点：

- 一个业务逻辑会分散在多个选项里
- 大组件里维护成本高

### Composition API

按“逻辑关注点”组织代码。  
同一块业务逻辑相关的状态、计算、监听、方法可以写在一起。

优点：

- 更适合复杂逻辑拆分
- 更适合封装 composable
- 更适合 TypeScript

所以两者的本质区别不是“语法风格不同”，而是“组织复杂度的方式不同”。

---

## Q15. 为什么 Vue 3 更推荐 Composition API？

**答：**

核心原因不只是“它更新”，而是它在复杂项目里更有工程价值。

### 1. 逻辑聚合更清晰

一个功能的状态、方法、watch、computed 可以写在一起，而不是散落在 `data`、`methods`、`watch` 各个块里。

### 2. 逻辑复用更自然

Vue 2 时代复用逻辑常依赖 mixin，但 mixin 容易出现：

- 命名冲突
- 来源不清
- 隐式依赖

Composition API 通过 composable 明确输入输出，复用成本更低。

### 3. TypeScript 体验更好

推导更自然，类型边界更清晰。

所以更推荐 Composition API，并不是因为 Options API 不能用，而是因为它更适合现代中大型项目。

## Q16. `<script setup>` 有什么优势？

**答：**

`<script setup>` 是 Vue 3 单文件组件中的编译时语法糖。

它的优势主要有：

- 样板代码更少
- 模板可直接使用顶层变量
- 更适合 Composition API
- TypeScript 体验更自然
- 宏能力如 `defineProps` / `defineEmits` 更简洁

它之所以成为主流，并不是“写法短一点”那么简单，而是因为它让 SFC 更贴近现代 Vue 的最佳实践。

如果面试官继续追问，可以补一句：`<script setup>` 是编译时处理，不是运行时多包了一层，所以性能上也没有额外运行时负担。

## Q17. 为什么直接解构 `reactive` 对象会丢失响应式？

**答：**

因为 `reactive` 的响应式依赖建立在“通过代理对象访问属性”这件事上。

```ts
const state = reactive({ count: 0 })
const { count } = state
```

这里解构出来的 `count` 已经是一个普通值，不再经过代理对象，自然也就失去了响应式链路。

正确做法通常是使用：

- `toRef`
- `toRefs`

这样可以把某个属性包装成仍与原对象保持连接的 ref。

这题本质考的是：你有没有理解 Vue 的响应式依赖不是“魔法全局同步”，而是依赖具体访问链路。

## Q18. `nextTick` 的作用是什么？

**答：**

Vue 的 DOM 更新通常是异步批量调度的，数据改了不代表 DOM 已经立刻是最新状态。

`nextTick` 的作用就是：等当前这一轮 DOM 更新完成后，再执行回调或继续后续逻辑。

常见场景：

- 更新数据后立刻读取最新 DOM
- 切换显示状态后去滚动定位
- 弹窗显示后自动聚焦输入框
- 列表渲染完后获取尺寸

所以 `nextTick` 不是“强制同步渲染”，而是“等待本轮异步更新落地后再继续”。

## Q19. Vue Router 中常见的导航守卫有哪些？

**答：**

常见守卫分为三类：

### 全局守卫

- `beforeEach`
- `beforeResolve`
- `afterEach`

### 路由独享守卫

- `beforeEnter`

### 组件内守卫

- 进入前、更新时、离开前相关能力

常见用途包括：

- 登录鉴权
- 权限校验
- 页面标题设置
- 埋点统计
- 阻止未保存表单直接离开

如果面试官追问“最常用哪个”，一般就是全局前置守卫 `beforeEach`。

## Q20. Pinia 和 Vuex 有什么区别？

**答：**

Pinia 是 Vue 3 主流状态管理方案，相比 Vuex，最大的差异在于：

### 1. API 更简洁

Pinia 不再强制区分 mutation 和 action，心智负担更低。

### 2. TypeScript 支持更自然

Pinia 的类型推导体验通常明显优于早期 Vuex。

### 3. 更符合 Composition API 风格

它更贴合 Vue 3 的整体使用习惯。

### 4. 组织方式更轻量

在中型项目里，Pinia 往往足够强又不显得繁琐。

如果是 Vue 3 新项目，通常优先 Pinia。

---

## Q21. 什么是自定义指令？适合什么场景？

**答：**

自定义指令适合封装底层 DOM 行为复用。

典型场景：

- 自动聚焦
- 点击外部关闭
- 拖拽
- 权限显示控制

为什么不是组件或 composable？

因为有些能力的本质是“对某个 DOM 元素做直接增强”，这种情况下指令比组件和逻辑函数更贴合。

但也要注意，不要把一切复用都做成指令。  
如果本质是状态逻辑复用，通常应该优先 composable。

## Q22. `keep-alive` 的作用是什么？

**答：**

`keep-alive` 用于缓存动态组件实例，避免切换时销毁和重建。

这意味着：

- 组件状态可以保留
- 重新切回来时不需要完整重建

典型场景：

- tab 页切换
- 表单中途切换保留输入内容
- 高成本页面缓存

同时要知道它会配合两个生命周期：

- `onActivated`
- `onDeactivated`

这题真正考的是你是否理解“缓存实例”和“普通条件渲染销毁重建”的区别。

---

## Q23. `provide` / `inject` 适合解决什么问题？

**答：**

它主要解决的是跨层级组件通信，避免中间层只是为了传值而不断透传 props。

典型场景：

- 表单上下文
- 主题配置
- 组件库内部共享状态
- 深层嵌套的业务上下文

它非常适合“树状上下文传递”，但不适合替代完整状态管理。

所以成熟判断是：

- 局部、层级型上下文共享：`provide/inject`
- 全局业务状态：Pinia

## Q24. Vue 性能优化有哪些常见手段？

**答：**

这题不要只罗列 API，要按优化思路回答。

### 1. 降低不必要渲染

- 合理拆分组件
- 使用 `computed`
- 避免模板中复杂计算
- 控制响应式粒度

### 2. 列表优化

- 稳定 `key`
- 大列表使用虚拟列表
- 避免整列表无意义重渲染

### 3. 加载优化

- 路由懒加载
- 异步组件
- 图片懒加载
- 缓存和 CDN

### 4. 特殊优化手段

- `v-once`
- `v-memo`
- `keep-alive`

但最后最好补一句：性能优化不是背技巧，而是先定位瓶颈，再做针对性处理。

## Q25. 你如何理解 Vue 在中大型项目中的优势与边界？

**答：**

我会分两部分说。

### 优势

- 学习曲线相对平滑
- 模板语法直观
- 官方生态集中，工程落地路径清晰
- Composition API 让复杂逻辑拆分和复用更自然
- 对中后台、管理系统、内容平台、运营类项目很友好

### 边界

- 如果团队缺少规范，组件通信和状态管理也会混乱
- 框架本身不能替代工程治理
- 真正的大型项目挑战，很多来自组织方式、目录设计、状态边界、类型约束，而不是 Vue 语法本身

所以更成熟的总结是：Vue 非常适合业务开发和中大型工程，但它的上限不只取决于框架，也取决于团队的工程能力和治理水平。
