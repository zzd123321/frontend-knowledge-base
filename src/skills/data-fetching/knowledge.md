# 数据请求与接口层核心知识点

前端开发里的“请求”绝不是 `fetch('/api')` 这么简单。真实项目里，请求层通常还会涉及：

- 接口设计风格
- 鉴权
- 错误处理
- 缓存
- 重试
- 分页
- 实时通信
- 数据一致性

所以学这一章，核心不是背几个协议名词，而是理解：

- 前端如何和服务端建立稳定的数据交互关系
- 不同通信方式适合什么场景
- 现代前端为什么越来越强调“请求层”和“服务端状态管理”

---

## 1. 什么是接口层

接口层可以理解为：

- 前端应用和后端能力之间的连接层

它不只是“发请求”，还要负责：

- 统一请求方式
- 统一鉴权
- 统一错误处理
- 统一数据格式适配
- 统一重试和超时策略

项目一大，请求层没有设计，业务代码通常会快速变乱。

---

## 2. 为什么请求层要单独设计

如果每个页面都自己写：

- 请求地址
- headers
- 错误处理
- loading 逻辑
- token 附带逻辑

很快会出现：

- 重复代码
- 处理不一致
- 很难统一升级

所以请求层设计的目标是：

- 把横向共性沉淀下来

---

## 3. REST 是什么

REST 是最常见的 Web API 风格之一。

它强调：

- 面向资源
- 用 HTTP 方法表达动作

例如：

- `GET /users`
- `GET /users/1`
- `POST /users`
- `PATCH /users/1`
- `DELETE /users/1`

REST 的优势在于：

- 简单直观
- 和 HTTP 语义天然贴合

---

## 4. REST 的优缺点

### 优点

- 易理解
- 工具生态成熟
- 缓存和 HTTP 语义好配合

### 缺点

- 复杂页面可能要调很多接口
- 返回结构容易冗余或不够灵活
- 前端容易遇到 over-fetching / under-fetching

这也是为什么很多复杂业务后来会考虑 GraphQL 或 BFF。

---

## 5. GraphQL 是什么

GraphQL 是一种查询语言和 API 运行时。

它的核心特点是：

- 前端声明自己要什么字段
- 服务端按这个结构返回

例如页面只需要用户的：

- `id`
- `name`
- `avatar`

就不必把整个用户对象全返回。

---

## 6. GraphQL 的优缺点

### 优点

- 查询更灵活
- 减少 over-fetching
- 强 schema 化
- 前端体验通常更好

### 缺点

- 缓存和权限设计更复杂
- 服务端实现复杂度更高
- 查询治理不当会有性能问题

所以 GraphQL 不是“比 REST 更高级”，而是：

- 更适合某些复杂数据聚合场景

---

## 7. BFF 和接口层的关系

BFF 是 Backend For Frontend。

它的核心思路是：

- 专门为前端或某个终端提供更贴合页面需求的后端聚合层

它能解决：

- 接口过碎
- 多端需求不同
- 前端请求瀑布过多

在很多复杂前端系统里，请求层设计和 BFF 设计是强相关的。

---

## 8. 请求生命周期

一次请求通常不只是“发出去然后拿到结果”。

它通常有这些状态：

- idle
- loading
- success
- error
- stale
- refreshing

理解这些状态非常重要，因为现代前端页面几乎都需要处理：

- 首次加载
- 错误兜底
- 后台刷新
- 手动重试

---

## 9. 请求封装的目标

一个好的请求封装通常会统一这些事情：

- baseURL
- headers
- token 附带
- 超时
- 错误格式处理
- 请求拦截
- 响应拦截

目标不是“炫封装”，而是：

- 把共性逻辑抽掉
- 让业务代码更聚焦

---

## 10. `fetch` 和 axios 的差异

### `fetch`

- 浏览器原生
- 更贴近底层
- 默认只在网络错误时 `reject`

### axios

- 封装更完整
- 拦截器和数据处理更方便
- 历史项目中很常见

两者都能用，关键在于：

- 项目是否需要统一封装能力

---

## 11. 错误处理为什么不能只靠 `catch`

请求错误通常分很多层：

- 网络错误
- 超时
- HTTP 状态码错误
- 业务错误码
- 数据格式异常

如果所有错误都只在页面里 `catch` 一下，很快会失控。

更好的设计是：

- 请求层统一处理可统一的问题
- 页面只处理业务语义相关的问题

---

## 12. 鉴权与请求层

请求层通常要负责附带身份信息。

常见方式：

- Cookie 自动携带
- `Authorization: Bearer xxx`

还要考虑：

- token 过期
- 刷新 token
- 401 统一处理
- 未登录跳转

这些逻辑如果散落在页面里，会很难维护。

---

## 13. 超时、取消与并发控制

真实项目里必须关注：

- 超时
- 取消请求
- 重复请求
- 并发数量

常见场景：

- 搜索联想要取消旧请求
- 页面切换要取消无效请求
- 列表筛选快速切换要只保留最后一次结果

请求层如果不考虑这些问题，体验和一致性都会很差。

---

## 14. 请求去重

请求去重指的是：

- 相同请求在短时间内不要重复发多次

它的价值：

- 降低服务端压力
- 避免浪费带宽
- 减少状态竞争

在服务端状态库里，这通常是内建能力的一部分。

---

## 15. 缓存为什么是请求层核心能力

请求层一旦脱离缓存，很多页面都会：

- 重复加载
- 首屏慢
- 切回来又重新转圈

缓存可以显著提升：

- 访问速度
- 交互连贯性
- 请求复用率

但缓存不是一开就完事，还要考虑：

- 何时过期
- 何时刷新
- 何时手动失效

---

## 16. 服务端状态缓存和浏览器缓存的区别

这是两个不同层次。

### 浏览器缓存

更偏 HTTP 和资源层。

### 服务端状态缓存

更偏应用层数据复用。

例如：

- `/api/users?page=1` 的结果在应用内 memory cache 中复用

这和浏览器基于响应头做缓存不是一回事。

---

## 17. SWR 思想是什么

SWR 可以理解为：

- 先返回旧缓存
- 再在后台重新验证和刷新

它的价值是：

- 页面先快
- 数据再新

这种思路非常适合：

- 用户可接受短暂旧数据
- 但仍希望尽快刷新到最新状态

---

## 18. TanStack Query / SWR 解决什么问题

这类库主要不是“帮你发请求”，而是在管理：

- 请求生命周期
- 缓存
- 去重
- 重试
- 失效
- 后台刷新

它们解决的是服务端状态管理问题。

这也是为什么现代前端越来越不建议：

- 把所有请求结果手动塞进普通全局 store

---

## 19. 重试策略

不是所有失败都应该立即重试。

更合理的判断是：

- 网络抖动是否值得重试
- 4xx 是否通常不该重试
- 5xx 是否可能适合重试
- 幂等请求是否更适合重试

还要注意：

- 指数退避
- 最大重试次数
- 用户可见反馈

---

## 20. 分页、无限滚动与游标

列表请求常见模式包括：

- 页码分页
- 无限滚动
- 游标分页

### 页码分页

简单直观，但在数据实时变化很强时可能不够稳。

### 游标分页

更适合大数据集和持续追加数据流。

前端要理解：

- 不同分页方式会影响缓存键、刷新策略和列表合并逻辑

---

## 21. 列表合并与去重

无限滚动或分页拼接时，经常会遇到：

- 重复项
- 顺序变化
- 更新后插入位置变化

所以请求层之外，列表数据合并本身也是重要逻辑。

如果设计不好，就容易：

- 出现重复数据
- 丢数据
- 刷新后顺序混乱

---

## 22. SSE 是什么

SSE 是 Server-Sent Events。

特点：

- 基于 HTTP
- 服务端到客户端单向推送

适合：

- 日志流
- 任务进度
- 消息通知流

如果只需要服务端持续推送文本事件流，SSE 往往比 WebSocket 更简单。

---

## 23. WebSocket 在接口层中的角色

WebSocket 适合：

- 聊天
- 实时协作
- 行情
- 高频实时通知

它和普通请求的关键区别在于：

- 连接建立后可以双向持续通信

因此它更像：

- 实时数据通道

而不是传统一次一答的接口。

---

## 24. 轮询与长轮询

### 轮询

客户端固定间隔反复问服务端有没有新数据。

### 长轮询

服务端尽量延迟响应，等有结果再返回。

它们都能解决“伪实时”问题，但成本和体验都不如真正适配场景的实时方案。

---

## 25. 数据一致性为什么难

请求层的一大难点是：

- 用户看到的数据、前端本地状态、服务端真实状态未必始终一致

典型问题：

- 列表页更新后详情页没同步
- 乐观更新失败没回滚
- 后台刷新覆盖用户本地修改
- 多标签页状态不一致

所以请求层设计必须考虑一致性，而不只是“能拿到数据”。

---

## 26. 数据失效与主动刷新

什么时候该认为缓存“过期”？

常见策略：

- 时间失效
- 用户触发刷新
- 某个 mutation 后关联查询失效
- 页面重新聚焦时刷新

好的数据层设计，不是永远缓存，也不是永远重新请求，而是：

- 在新鲜度和性能之间找到平衡

---

## 27. Mutation 和 Query 的区别

在现代服务端状态库语境里：

- Query 更偏读取
- Mutation 更偏写入

写入成功后，往往还要考虑：

- 更新本地缓存
- 使某些查询失效
- 是否乐观更新

这也说明请求层不是“只有 GET”，而是完整的数据交互体系。

---

## 28. 统一数据格式适配

很多后端接口风格不完全一致。

例如：

- 字段命名不统一
- 错误结构不同
- 分页结构不同

如果每个页面自己适配，会很乱。

更合理的做法是：

- 在接口层或数据层统一做适配

这样业务层会更干净。

---

## 29. Mock 和接口联调

请求层设计也会影响开发协作。

常见协作方式：

- 本地 mock
- MSW
- 假数据层
- 联调环境切换

如果接口层抽象清晰：

- mock、测试、联调都会容易很多

---

## 30. 数据请求层的最佳实践

- 请求逻辑和页面逻辑分层
- 统一鉴权和错误处理
- 明确缓存键和失效策略
- 区分 query 与 mutation
- 把服务端状态单独对待
- 优先设计一致性，再谈“封装得优不优雅”

---

## 31. 常见误区

- 把所有请求结果都塞进普通全局 store
- 页面里直接散落大量请求细节
- 不区分网络错误和业务错误
- 不做取消和去重
- 不考虑缓存失效
- 盲目乐观更新

这些问题会让数据层越来越难维护。

---

## 32. 数据请求与接口层最值得掌握的主线

如果只抓一条主线，就抓这条：

1. 前端如何从服务端稳定获取数据
2. 不同接口风格适合什么场景
3. 请求层如何统一鉴权、错误、超时、取消和适配
4. 服务端状态如何缓存、失效、刷新和保持一致
5. 实时通信、分页、乐观更新如何纳入统一数据体系

把这条主线建立起来，你就不会再把请求层理解成“发个接口”，而会把它当成前端架构的重要基础。
