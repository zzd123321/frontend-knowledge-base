# 前端测试核心知识点

前端测试不是为了追求“覆盖率好看”，而是为了降低回归风险、提升重构信心、保证关键业务长期可演进。真正成熟的测试体系，通常不是某一种测试特别多，而是会把静态检查、单元测试、组件测试、集成测试、端到端测试组合起来，分别覆盖不同层面的风险。

这一章最值得建立的认知是：

- 不同测试层级解决的是不同问题
- 测试越贴近真实用户，价值通常越高，但成本也越高
- 好测试关注行为和结果，不盯实现细节
- 测试本身也是工程体系的一部分，需要可维护、可运行、可持续

---

## 1. 为什么前端需要测试

测试的核心价值不是“替代人工”，而是把本来靠经验和记忆防守的问题，变成自动化校验。

常见收益：

- 提前发现逻辑错误
- 降低重构风险
- 稳定核心业务流程
- 提升多人协作信心
- 帮助沉淀系统行为文档

前端尤其需要测试，因为前端系统往往同时包含：

- 纯逻辑
- 组件渲染
- 用户交互
- 网络请求
- 浏览器 API
- 路由、状态管理、权限控制

这些层之间耦合一旦变复杂，只靠手测很难长期稳定。

---

## 2. 什么是高价值测试

高价值测试通常具备几个特点：

- 能及时发现真实缺陷
- 稳定，不容易随机失败
- 易读，看到测试名就知道它在保护什么
- 易改，业务调整时不需要大面积重写
- 跑得足够快，能进入日常开发流程

反过来说，低价值测试常见表现是：

- 断言一堆实现细节
- 改一行文案就挂
- 快照巨大但没人看
- 需要大量手工维护
- 不能说明真实业务是否可用

---

## 3. 测试分层思维

前端测试通常按层次理解：

- 静态检查
- 单元测试
- 组件测试
- 集成测试
- E2E 测试

它们的区别，不只是“跑在什么环境”，更重要的是：

- 被验证对象是什么
- 能覆盖哪些风险
- 运行成本是多少
- 失败后定位难度多大

一个常见误区是把所有测试都推到 E2E。这样虽然“像真人”，但执行慢、维护重、定位难，最后往往跑不起来。

---

## 4. 测试金字塔与测试奖杯

### 测试金字塔

经典金字塔强调：

- 底层单元测试多
- 中层集成测试适量
- 顶层 E2E 少而精

它想解决的问题是：

- 保持反馈速度
- 控制测试成本
- 让大部分问题在更低层暴露

### 测试奖杯

近年更常见的观点是 Testing Trophy，强调：

- 静态检查
- 集成测试
- 少量单元测试
- 少量 E2E

它强调“更贴近真实使用方式的测试”价值更高。

真正落地时不必教条站队，关键是：

- 低层测试负责兜逻辑
- 中层测试负责兜交互与模块协作
- 高层测试负责兜主流程

---

## 5. 静态检查也是测试体系的一部分

很多问题甚至不用运行代码就能发现。

常见静态保障：

- TypeScript
- ESLint
- 样式检查
- 构建校验
- 提交规范检查

它们的作用是：

- 提前拦截明显错误
- 保持代码风格一致
- 降低低级问题进入运行时的概率

所以一个成熟项目的质量体系，不会把“测试”只理解成 `test` 命令。

---

## 6. 单元测试

单元测试关注最小可测试单元，通常是：

- 工具函数
- 纯函数
- 业务规则
- 数据转换逻辑
- 边界判断逻辑

适合单元测试的代码特征：

- 输入明确
- 输出明确
- 依赖少
- 副作用可控

例如：

```ts
function formatPrice(value: number) {
  return `¥${value.toFixed(2)}`
}
```

这类函数很适合用单元测试覆盖正常值、边界值、异常值。

---

## 7. 单元测试该测什么

优先测这些内容：

- 条件分支
- 边界情况
- 特殊输入
- 错误处理
- 容易回归的业务规则

不太值得在单元层面反复测的内容：

- 第三方库本身已经保证的行为
- 简单 getter / setter
- 没有分支、没有风险的样板代码

衡量标准不是“这段代码能不能测”，而是“测它值不值”。

---

## 8. 组件测试

组件测试关注组件在给定输入下如何渲染，以及用户交互后行为是否正确。

重点通常包括：

- 是否正确渲染
- 是否响应用户操作
- 状态变化是否反映到界面
- props 变化是否产生正确结果
- 错误态、空态、加载态是否完整

一个好的组件测试，往往更像“模拟用户使用组件”，而不是“检查组件内部怎么实现”。

---

## 9. 组件测试与单元测试的边界

单元测试更关注：

- 逻辑最小单元
- 输入输出
- 内部算法

组件测试更关注：

- 组件渲染结果
- 用户交互过程
- DOM 上的可见结果

举个例子：

- `calculateDiscount()` 适合单元测试
- “点击加入购物车按钮后按钮变成 loading，成功后数量增加” 更适合组件测试

---

## 10. 集成测试

集成测试关注多个模块组合后的行为。

常见场景：

- 组件 + 路由
- 组件 + store
- 表单 + 校验 + 提交
- 页面 + 请求 + 状态切换
- 弹窗 + 权限 + 业务流程

集成测试的价值很高，因为很多真实问题并不是某一个函数错了，而是模块拼起来以后出现问题。

---

## 11. E2E 测试

E2E 测试从真实用户视角验证完整流程。

比如：

1. 打开登录页
2. 输入用户名和密码
3. 点击登录
4. 跳转首页
5. 校验欢迎信息

它最擅长保护的是：

- 核心主流程
- 路由跳转
- 真实浏览器行为
- 多页面联动
- 线上最容易出事故的关键路径

常见工具：

- Playwright
- Cypress

---

## 12. 不同层级测试的取舍

可以粗略这样理解：

- 单元测试：快、便宜、定位准，但离真实使用稍远
- 组件/集成测试：平衡最好，前端项目通常最值得投入
- E2E 测试：最真实，但慢、重、脆弱一些

所以更合理的策略通常是：

- 用单元测试守住纯逻辑
- 用组件/集成测试守住业务交互
- 用少量 E2E 守住关键主流程

---

## 13. 常见测试工具

### 单元与组件测试

- Vitest
- Jest

### DOM 交互测试

- Testing Library

### 网络层 Mock

- MSW

### E2E

- Playwright
- Cypress

现代前端里较常见的组合是：

- Vitest + Testing Library + MSW
- Playwright 负责关键 E2E

---

## 14. 断言库与测试运行器

测试运行器负责：

- 发现测试文件
- 运行测试
- 输出结果
- 提供 Mock、Fake Timer 等能力

断言库负责：

- 判断结果是否符合预期

比如：

```ts
expect(result).toBe(3)
expect(button).toBeDisabled()
```

很多工具会把运行器和断言能力打包到一起，但理解职责边界依然重要。

---

## 15. 测试的结构写法

一个清晰的测试通常遵循 AAA：

- Arrange：准备数据和环境
- Act：执行操作
- Assert：断言结果

示例：

```ts
test('adds item after click', async () => {
  render(<Cart />)

  await user.click(screen.getByRole('button', { name: '加入购物车' }))

  expect(screen.getByText('购物车(1)')).toBeInTheDocument()
})
```

这种结构可读性强，也更方便后续维护。

---

## 16. 测试命名

好测试名应该表达：

- 场景
- 条件
- 结果

例如：

- `should render empty state when list is empty`
- `submits form successfully after valid input`
- `shows error message when request fails`

不推荐：

- `test1`
- `works`
- `renders correctly`

越到后期，测试名越像团队的行为文档。

---

## 17. 用户视角测试

Testing Library 的核心思想是：

- 越接近用户使用方式的测试越可靠

因此优先查询：

- 文本
- role
- label
- placeholder
- 可访问名称

例如优先使用：

- `getByRole`
- `getByLabelText`
- `getByText`

而不是优先使用：

- class 名
- data-testid
- 内部组件实例

---

## 18. 为什么要少测实现细节

实现细节测试通常很脆弱。

例如下面这些就容易过度绑定实现：

- 断言组件内部 state
- 断言某个私有方法被调用次数
- 断言某个 class 名刚好等于某个字符串
- 断言内部 DOM 结构一层不变

问题是：

- 业务没变，测试却挂
- 测试阻碍重构
- 代码被测试“绑架”

更好的方式是测最终行为和界面结果。

---

## 19. Mock、Stub、Spy、Fake 的区别

这些概念经常被混用，但最好分清：

- Mock：替代真实依赖，并预设行为
- Stub：提供简单替身返回固定结果
- Spy：监听函数是否被调用、怎么被调用
- Fake：用一个轻量可运行实现替换真实对象

在前端日常里，大家常把这些统称为 Mock，但理解区别有助于选更合适的测试方式。

---

## 20. 什么情况下需要 Mock

常见需要 Mock 的对象：

- 网络请求
- 时间
- 随机数
- 路由
- 浏览器 API
- 大型第三方依赖

需要 Mock 的原因通常是：

- 不想依赖外部服务
- 需要构造稳定可控场景
- 需要验证边界条件
- 某些依赖在测试环境不存在

---

## 21. 什么时候不该过度 Mock

过度 Mock 的风险很大：

- 测试环境和真实环境偏差太大
- 集成问题被掩盖
- 测出来“都通过”，上线却坏

尤其不建议把整个组件树、整个请求层、整个状态层都 mock 成空壳后再说“测试通过”。

一般原则：

- 越靠近系统边界越适合 Mock
- 越接近核心业务组合越应该保留真实交互链路

---

## 22. 网络请求测试

前端测试里，请求是最常见的依赖。

常见做法：

- mock `fetch`
- mock `axios`
- 使用 MSW 从网络层模拟

其中 MSW 的价值在于：

- 模拟更接近真实请求行为
- 不强耦合某个请求库
- 组件、页面、E2E 都能复用一套接口假数据思路

---

## 23. MSW 的价值

MSW 不是简单“伪造函数返回值”，而是在请求层拦截。

这让测试更接近真实场景：

- 组件仍然像平时一样发请求
- 可以模拟成功、失败、超时、空数据
- 不需要侵入业务代码

因此在组件测试和集成测试里，MSW 通常比直接 mock 某个 API 方法更稳。

---

## 24. 异步测试

前端测试大量涉及异步：

- 请求返回
- 状态更新
- DOM 更新
- 定时器
- 动画

异步测试的关键不是“加个 `await` 就行”，而是明确你在等什么。

常见等待对象：

- 元素出现
- 元素消失
- 请求结束
- 文案变化
- 按钮可用状态变化

---

## 25. 异步测试常见写法

```ts
await screen.findByText('加载完成')

await waitFor(() => {
  expect(mockFn).toHaveBeenCalled()
})
```

原则上：

- 能用 `findBy*` 就优先用 `findBy*`
- 只有需要等待某个复杂副作用时才用 `waitFor`

不要盲目大量使用：

- `setTimeout`
- 人工 sleep

因为它们会让测试变慢且不稳定。

---

## 26. Fake Timers

Fake Timer 适合测试这些逻辑：

- debounce
- throttle
- timeout
- interval
- 倒计时

例如：

```ts
vi.useFakeTimers()
vi.advanceTimersByTime(300)
```

好处：

- 测试更快
- 不用真的等 3 秒
- 可精确控制时间推进

注意：

- 使用后记得恢复真实计时器
- 和 Promise、微任务一起用时要理解调度顺序

---

## 27. 浏览器环境与测试环境

前端测试并不总是在真实浏览器里跑。

常见环境：

- Node
- jsdom
- 真浏览器

区别在于：

- Node 没有 DOM
- jsdom 模拟了大量 DOM API，但不等于真正浏览器
- 真浏览器最真实，但运行成本更高

因此：

- 单元和多数组件测试常跑在 jsdom
- E2E 常跑在真实浏览器

---

## 28. jsdom 的边界

很多人以为 jsdom 就等于浏览器，其实不是。

它不擅长完整模拟：

- 真实布局计算
- 完整渲染流水线
- 某些媒体能力
- 某些原生 API

所以遇到这些场景：

- 拖拽
- 复杂动画
- 真实尺寸依赖
- 浏览器兼容问题

就更适合上更高层测试。

---

## 29. 组件测试中的交互模拟

交互模拟要尽量贴近真实用户。

常见操作：

- 点击
- 输入
- 清空
- 聚焦/失焦
- 键盘操作
- 选择下拉项

相比直接触发底层事件，更推荐使用用户层 API，因为它们会更真实地串起多个事件过程。

例如一次输入，不只是 `change`，可能还涉及：

- focus
- keydown
- input
- keyup
- change

---

## 30. 表单测试

表单是前端最典型、也最容易出回归的区域。

应该重点覆盖：

- 必填校验
- 格式校验
- 提交按钮状态
- 请求中 loading
- 成功提示
- 失败提示
- 重复提交防护

表单测试不要只测“按钮能点”，而要覆盖完整的用户路径。

---

## 31. 路由与页面级测试

页面测试通常会涉及：

- 路由跳转
- 参数解析
- 权限判断
- 页面数据加载

适合测试的问题：

- 未登录是否跳登录
- 错误路由是否进入 404
- query 参数变化是否触发刷新
- 页面切换是否正确保留状态

这类测试通常已经接近集成测试层级。

---

## 32. 状态管理测试

状态管理相关测试通常有三种思路：

- 单独测 store / reducer / action
- 在组件中带真实 store 跑集成测试
- 在页面流程里通过 UI 验证状态变化

更推荐的思路通常是：

- 纯逻辑层单测
- 组件和页面尽量用真实 store 集成

这样更能防止“store 单独没问题，但接到页面上就出问题”。

---

## 33. 错误态与边界态测试

很多项目测试最大的问题不是“完全没测”，而是只测了 happy path。

一定要重点补的内容：

- 空数据
- 请求失败
- 权限不足
- 加载中
- 超时
- 重试
- 表单校验失败

真正线上更容易出事故的，常常正是这些边界场景。

---

## 34. 快照测试

快照测试的优点：

- 上手快
- 能快速发现输出变化

缺点：

- 很容易产生低价值快照
- 一旦快照巨大，团队会机械更新而不审查
- 不能替代有语义的断言

更合理的使用方式：

- 只对稳定、结构明确的小型输出用快照
- 关键行为仍然写显式断言

---

## 35. 覆盖率怎么看

常见覆盖率指标：

- Statements
- Branches
- Functions
- Lines

覆盖率有价值，但不能被神化。

高覆盖率不一定代表测试有效，因为可能存在：

- 只执行到了代码，没有断言关键结果
- 只跑到了 happy path
- 大量低价值快照或样板测试

更好的看法是：

- 覆盖率是风险提示器
- 不是质量本身

---

## 36. Flaky Test 是什么

Flaky Test 指的是：

- 代码没改
- 测试有时过、有时不过

这类测试危害很大，因为它会逐渐摧毁团队对测试结果的信任。

常见原因：

- 异步等待不稳定
- 共享状态污染
- 依赖真实时间
- 依赖真实网络
- 选择器脆弱
- 环境初始化不一致

Flaky Test 不是“小毛病”，而是应该优先治理的质量问题。

---

## 37. 如何避免脆弱测试

尽量做到：

- 用语义查询而不是 DOM 结构查询
- 不依赖固定时间等待
- 不依赖接口真实返回
- 每个测试独立初始化状态
- 清理副作用
- 让断言和业务目标直接对应

一句话：

- 测试应该保护业务，而不是绑死实现

---

## 38. 测试目录与组织方式

常见组织方式有两种：

- 按功能模块 colocate，在源码旁边放测试
- 按测试目录集中管理

两种都可以，关键是统一。

通常一个测试文件内建议：

- 同一模块或同一组件放在一起
- `describe` 描述上下文
- 单个 `test` 聚焦单一行为

不要一个测试同时断言十几件事情，失败后很难定位。

---

## 39. beforeEach / afterEach 的使用

这些钩子适合：

- 重置 mock
- 清理 DOM
- 重建测试环境
- 恢复全局对象

但不要滥用。

如果某些准备逻辑藏得太深，会让测试可读性变差。一个经验是：

- 通用初始化放钩子
- 和当前测试强相关的准备直接写在测试体里

---

## 40. CI 中的测试

测试的真正价值，不只是本地能跑，而是进入持续集成流程。

CI 常见做法：

- PR 必跑单元/组件测试
- 主分支定时跑更完整测试
- E2E 覆盖关键流程
- 失败即阻断合并

进入 CI 后要特别关注：

- 运行速度
- 环境一致性
- 测试数据稳定性
- 截图、录像、日志等调试信息

---

## 41. E2E 测试最佳实践

E2E 最好只覆盖关键主路径：

- 登录
- 下单
- 支付
- 权限切换
- 表单提交流程
- 关键后台管理操作

注意：

- 不要试图用 E2E 覆盖所有逻辑
- 不要让 E2E 成为唯一测试手段
- 尽量保证测试数据可控和隔离

E2E 的任务是守住“系统能不能真正用”，不是代替所有低层测试。

---

## 42. 可访问性测试

前端测试不应该只测“能不能用”，也要关注“是不是所有人都能用”。

常见可测点：

- 是否存在正确 role
- 表单是否有 label
- 键盘是否可操作
- 焦点是否合理移动
- 错误提示是否可感知

这也是为什么 Testing Library 很强调基于可访问语义查询。

---

## 43. 视觉回归测试

有些问题逻辑没错，但界面错了。

比如：

- 布局错位
- 弹层遮挡
- 主题色丢失
- 组件样式回退

这种问题可通过视觉回归测试补充。

它适合：

- 设计要求稳定的页面
- 组件库
- 营销页

但也要注意：

- 视觉测试对环境和基线管理要求更高

---

## 44. 合同测试与接口约定

前后端分离项目里，还有一种很实用的思路是合同测试。

重点不是测“接口通不通”，而是测：

- 返回字段结构是否符合约定
- 字段类型有没有变
- 错误码格式是否稳定

这能减少“前端按旧结构写，后端悄悄改了”的问题。

---

## 45. 测试策略该怎么设计

比较实用的一套策略通常是：

- 工具函数做单元测试
- 关键组件做组件测试
- 页面级交互做集成测试
- 核心业务流程做 E2E
- 静态检查前置进提交和 CI

也就是说：

- 不求所有代码都测
- 先测高风险、高价值、高复用部分

---

## 46. 什么时候该补测试

最值得优先补测试的地方：

- 老代码准备重构前
- 最近频繁出 bug 的模块
- 业务核心链路
- 通用组件和工具函数
- 权限、支付、表单、数据转换这类高风险逻辑

如果资源有限，先补这些位置，收益往往最高。

---

## 47. 常见误区

- 追求覆盖率而忽略测试价值
- 只测 happy path
- 严重依赖实现细节
- 把所有东西都 mock 掉
- 试图只靠 E2E 解决所有问题
- 测试写完后从不治理 flaky 问题

这些问题一旦积累，测试会从“质量资产”变成“协作负担”。

---

## 48. 前端测试最值得掌握的主线

如果只抓一条主线，就抓这条：

1. 明确系统有哪些风险
2. 用不同层级测试覆盖不同风险
3. 优先测试用户真正关心的行为
4. 把测试融入日常开发和 CI
5. 让测试成为重构和迭代的保护网

做到这几点，测试就不再是额外负担，而会真正变成前端工程稳定性的基础能力。
