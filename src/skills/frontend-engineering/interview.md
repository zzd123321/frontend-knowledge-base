# 前端工程化进阶面试题

## Q1. 什么是前端工程化？

**答：**

前端工程化可以理解为：

- 用工具、规范和流程，让前端开发从个人写代码变成可协作、可持续演进的工程体系

它不仅包括构建工具，还包括：

- 项目结构
- 代码规范
- 测试
- CI/CD
- 发布流程
- 监控反馈

所以更成熟的回答不是只说 Webpack、Vite，而是从研发全链路去看。

---

## Q2. 前端工程化通常包含哪些部分？

**答：**

常见包括：

- 项目结构设计
- Lint / Format / Type Check
- 构建与打包
- 测试体系
- Git 协作流程
- CI/CD
- 环境管理
- 监控与发布治理

这些部分共同作用，目标是提升协作效率和系统稳定性。

---

## Q3. 为什么说工程化不是“工具越多越好”？

**答：**

因为工程化的目标是解决真实问题，而不是堆配置。

如果工具很多，但：

- 没人真正执行
- 流程太重
- 成本大于收益

那它反而会拖慢团队。

所以工程化应该强调：

- 合适
- 自动化
- 可持续

---

## Q4. CI 和 CD 分别是什么？

**答：**

### CI

持续集成，强调代码提交后自动做校验，比如：

- Lint
- 测试
- 构建

### CD

持续交付 / 持续部署，强调把通过校验的代码稳定发布到环境中。

面试里最好补一句：

- CI 关注“代码合进来有没有问题”
- CD 关注“代码怎么稳定交付出去”

---

## Q5. Monorepo 的优缺点是什么？

**答：**

### 优点

- 多包共享代码方便
- 统一依赖和规范
- 跨包改动更直接

### 缺点

- 依赖管理复杂
- CI 成本可能更高
- 构建和测试治理要求更高

所以 Monorepo 更适合需要共享基础设施和组件的团队，但前提是工程治理能力要跟上。

---

## Q6. commit 规范和 Git Hooks 为什么有价值？

**答：**

它们的价值在于：

- 保持提交历史有语义
- 提前拦截低级问题
- 降低 review 成本
- 便于自动生成 changelog

例如在提交前自动跑 lint、类型检查，就能把很多问题挡在进入仓库之前。

---

## Q7. Feature Flag 和灰度发布解决什么问题？

**答：**

它们解决的是上线风险控制问题。

### Feature Flag

让功能是否开启变成可控开关。

### 灰度发布

让新版本先面向一部分用户开放。

这样可以做到：

- 小范围验证
- 快速止损
- 降低全量发布风险

---

## Q8. 为什么监控和埋点也属于工程化的一部分？

**答：**

因为工程化不是上线结束，而是要形成闭环。

监控和埋点可以帮助团队知道：

- 哪些错误在线上发生
- 性能是否退化
- 用户行为和转化是否变化

没有这些反馈，工程化就只能做到“能发布”，做不到“能持续稳定运营”。

---

## Q9. 多环境管理时最容易踩什么坑？

**答：**

常见坑包括：

- 环境变量命名混乱
- 测试和生产配置不一致
- 把敏感信息直接暴露到前端
- 构建时变量和运行时配置混淆

所以多环境管理的关键不是文件越多越好，而是边界要清楚、规则要统一。

---

## Q10. 前端工程化进阶的核心目标是什么？

**答：**

一句话总结就是：

- 让前端项目在多人协作、持续迭代和长期维护中仍然保持高质量和高效率

更具体地说，是让：

- 代码结构可持续
- 质量检查自动化
- 发布流程稳定
- 问题可追踪和可回滚

这才是工程化真正的价值。
