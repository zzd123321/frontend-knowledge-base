# 浏览器 API 面试题

## Q1. DOM 和 BOM 有什么区别？

**答：**

DOM 是文档对象模型，核心关注的是页面结构，也就是 HTML 被浏览器解析后的节点树。我们操作元素、修改文本、增删节点，本质上都属于 DOM 范畴。

BOM 是浏览器对象模型，核心关注的是浏览器窗口环境本身，而不是页面结构。常见 BOM 对象包括：

- `window`
- `location`
- `history`
- `navigator`
- `screen`

所以更准确地说：

- DOM 管“页面内容和结构”
- BOM 管“浏览器窗口和宿主环境能力”

实际开发中二者经常一起出现，比如你先通过 DOM 找到元素，再通过 BOM 读取地址或控制历史记录。

---

## Q2. `querySelector` 和 `getElementById` 有什么区别？

**答：**

### `getElementById`

- 只能通过 id 查找
- 返回单个元素
- 语义直接，性能通常也比较直接

### `querySelector`

- 支持完整 CSS 选择器
- 更灵活
- 可以查类、属性、层级关系等复杂结构

所以如果只是根据 id 取一个元素，`getElementById` 很清晰；如果你需要复杂选择器能力，`querySelector` 更常用。

现代项目中，很多人更偏向统一使用 `querySelector` / `querySelectorAll`，因为 API 更一致、表达力更强。

## Q3. `innerHTML`、`innerText`、`textContent` 有什么区别？

**答：**

这题是浏览器 API 非常高频的细节题。

### `innerHTML`

返回或设置元素内部的 HTML 字符串。  
如果你给它赋值，浏览器会解析字符串并生成对应 DOM。

优点是灵活，缺点是：

- 有性能成本
- 存在 XSS 风险

### `innerText`

更接近“用户真正看到的文本”，会考虑样式和隐藏元素的影响。  
某些情况下读取它会触发布局相关计算，因此性能上可能不如 `textContent` 稳定。

### `textContent`

返回纯文本内容，不解析 HTML，也不太关注视觉表现。  
通常更稳定、更高效，也更安全。

实际开发里：

- 如果只想处理纯文本，优先 `textContent`
- 如果要插入 HTML 结构，才考虑 `innerHTML`

## Q4. 事件冒泡和事件捕获是什么？

**答：**

浏览器事件流通常分三个阶段：

1. 捕获阶段
2. 目标阶段
3. 冒泡阶段

### 捕获

事件从最外层往目标元素传播。

### 目标

事件到达真正触发的目标元素。

### 冒泡

事件从目标元素再向外层元素逐级冒泡。

默认情况下，`addEventListener` 绑定的监听器通常是在冒泡阶段触发；如果你传 `true` 作为第三个参数，则会在捕获阶段监听。

这题最重要的是理解：事件不是“只在目标元素发生”，而是沿着 DOM 树传播，这也正是事件委托成立的基础。

---

## Q5. 什么是事件委托？为什么它有用？

**答：**

事件委托是指把本来应该绑定在多个子元素上的事件，统一绑定到它们共同的父元素上，再利用事件冒泡判断具体是哪个子元素触发了事件。

比如一个列表中有很多 `li`，你可以不在每个 `li` 上都绑定 click，而是只在 `ul` 上绑定一次。

它的好处包括：

- 减少事件监听器数量
- 降低内存与管理成本
- 对动态新增节点天然友好

适合场景：

- 列表
- 表格
- 菜单项
- 动态渲染内容

所以事件委托不只是一个“优化技巧”，它其实是基于浏览器事件传播模型的一种非常实用的设计方式。

## Q6. `target` 和 `currentTarget` 的区别是什么？

**答：**

### `target`

表示真正触发事件的元素。

### `currentTarget`

表示当前正在执行这个事件处理函数的元素，也就是监听器绑定在哪个元素上，它通常就是哪个元素。

事件委托里这两个概念特别重要：

- 你通常在父元素上绑定监听器，所以 `currentTarget` 是父元素
- 但真正被点到的可能是子元素，所以 `target` 是子元素

如果能把这两个概念说清楚，说明你对事件传播不只是停留在“会写 click”层面。

## Q7. `preventDefault` 和 `stopPropagation` 的区别是什么？

**答：**

### `preventDefault()`

阻止浏览器默认行为。  
比如：

- 阻止表单提交
- 阻止 a 标签跳转

### `stopPropagation()`

阻止事件继续向上或向下传播。  
它不影响默认行为，只影响传播链路。

这题很容易被答混。  
最准确的理解是：

- `preventDefault` 管“默认行为”
- `stopPropagation` 管“传播路径”

---

## Q8. `DOMContentLoaded` 和 `load` 的区别是什么？

**答：**

### `DOMContentLoaded`

当 HTML 文档被完全解析，DOM 树构建完成后就会触发。  
它不要求图片、样式、iframe 等所有资源都加载完。

### `load`

要等整个页面的资源，包括图片、CSS、脚本等全部加载完成后才触发。

因此：

- 只关心 DOM 是否可操作，通常看 `DOMContentLoaded`
- 关心整页资源是否完全准备好，才看 `load`

---

## Q9. `localStorage`、`sessionStorage`、`cookie` 有什么区别？

**答：**

这题建议从“生命周期”“是否自动携带”“容量与用途”几个维度回答。

### `localStorage`

- 持久存储
- 同源共享
- 除非手动删除，否则长期存在
- 不会自动随请求发送

### `sessionStorage`

- 会话级存储
- 关闭当前标签页后通常失效
- 不会自动随请求发送

### `cookie`

- 体积较小
- 会按规则自动随请求发送到服务端
- 可配置过期时间、路径、域、SameSite 等属性

所以：

- 如果是纯前端本地状态保存，`localStorage/sessionStorage` 更常见
- 如果需要和服务端会话机制结合，Cookie 更常见

---

## Q10. 什么时候用 IndexedDB？

**答：**

当你需要在浏览器本地保存大量结构化数据时，IndexedDB 比 `localStorage` 更合适。

典型场景包括：

- 离线缓存
- 大量列表数据持久化
- 草稿系统
- 文件元数据
- 本地数据库能力

为什么不用 `localStorage`？

因为 `localStorage`：

- 容量有限
- API 同步阻塞
- 不适合复杂结构和大数据量

而 IndexedDB：

- 更适合大规模结构化数据
- 异步 API
- 更接近浏览器内数据库

## Q11. `requestAnimationFrame` 和 `setTimeout` 的区别是什么？

**答：**

### `setTimeout`

是通用定时任务工具，到了指定时间后把回调加入任务队列。  
它不关心当前浏览器是否准备好渲染，也不保证和屏幕刷新节奏同步。

### `requestAnimationFrame`

专门为动画场景设计。  
浏览器会在下一次重绘前调用回调，因此它更适合：

- 动画
- 平滑滚动
- 高频视觉更新

它的优势在于：

- 更容易和刷新节奏对齐
- 页面隐藏时通常会自动降频

所以如果是动画场景，优先 `requestAnimationFrame`；如果只是普通延迟任务，才考虑 `setTimeout`。

---

## Q12. 为什么频繁操作 DOM 会影响性能？

**答：**

因为 DOM 操作不仅仅是“改一个值”。  
很多 DOM 操作可能触发：

- 样式重新计算
- 布局计算
- 重绘
- 合成

尤其是当你频繁交替做“读取布局信息”和“修改样式”时，很容易造成布局抖动和性能下降。

另外，DOM 是浏览器宿主对象，操作它通常比纯 JavaScript 数据计算更贵。

所以优化的核心思路一般是：

- 批量更新 DOM
- 减少布局读取
- 避免频繁触发回流

---

## Q13. `getBoundingClientRect` 有什么用？

**答：**

`getBoundingClientRect()` 可以获取元素相对于视口的位置和尺寸信息，比如：

- `top`
- `left`
- `width`
- `height`

常见用途包括：

- 懒加载判断
- 吸顶效果
- 弹层定位
- 拖拽计算
- 滚动可视区判断

它是定位和尺寸测量类功能里非常常用的 API。

---

## Q14. `postMessage` 使用时要注意什么？

**答：**

`postMessage` 常用于：

- iframe 通信
- 不同窗口通信
- Worker 通信

它本身很好用，但最大的注意点是安全。

### 发送时

尽量指定明确的目标源，而不是随手写 `*`。

### 接收时

必须校验：

- `event.origin`
- 必要时校验消息结构

因为跨窗口消息本质上也是外部输入，如果不校验来源，就可能被恶意页面伪造消息攻击。

## Q15. WebSocket 和 HTTP 轮询有什么区别？

**答：**

### HTTP 轮询

客户端定时向服务端询问有没有新数据。

问题在于：

- 请求开销大
- 实时性一般
- 很多请求可能都是空结果

### WebSocket

建立连接后可以双向持续通信。

优点：

- 服务端可主动推送
- 实时性更好
- 更适合高频消息场景

所以两者区别的核心不是“一个新一个旧”，而是：

- 轮询是反复问
- WebSocket 是建立通道后持续交流

---

## Q16. `IntersectionObserver` 适合什么场景？

**答：**

`IntersectionObserver` 用来观察元素是否进入视口或与某个根容器相交。

典型场景：

- 图片懒加载
- 无限滚动
- 曝光埋点
- 元素进入可视区时触发动画

它相比手写 `scroll + getBoundingClientRect` 的优势在于：

- 语义更清晰
- 浏览器层面优化更好
- 避免你自己在 scroll 高频事件里做大量计算

---

## Q17. `ResizeObserver` 的作用是什么？

**答：**

`ResizeObserver` 用于监听元素尺寸变化，而不是窗口尺寸变化。

这很重要，因为有些布局变化并不是用户 resize 窗口导致的，而是：

- 父容器变化
- 内容撑开
- 布局切换

典型场景：

- 图表重绘
- 自适应组件
- 动态容器布局

---

## Q18. Web Worker 的意义是什么？

**答：**

Web Worker 的核心意义是把耗时计算放到独立线程执行，避免阻塞主线程。

适合场景：

- 大数据计算
- 文件解析
- 图像处理
- 大量 JSON 转换

但要注意它的边界：

- Worker 不能直接操作 DOM
- 主线程和 Worker 要通过消息通信

所以 Worker 适合“计算密集型”任务，不适合“直接操纵页面”任务。

---

## Q19. 为什么 `innerHTML` 有安全风险？

**答：**

因为 `innerHTML` 会把字符串当作 HTML 解析并插入页面。  
如果这个字符串来自不可信输入，就可能包含恶意脚本、危险标签或事件处理逻辑，从而引发 XSS。

所以在实际开发里：

- 非必要不要用 `innerHTML`
- 如果必须用，就要确保内容经过严格清洗

这题的本质不是 API 使用细节，而是前端安全意识。

---

## Q20. 浏览器 API 使用中最常见的内存泄漏来源有哪些？

**答：**

常见来源包括：

- 事件监听未移除
- 定时器未清理
- Observer 未断开
- WebSocket 或订阅未关闭
- 闭包长期持有大对象
- DOM 节点被移除但引用仍然保留

这题最重要的是体现你有“资源生命周期管理”的意识。  
在现代前端框架中，这些问题通常会落到组件卸载清理逻辑里。
