## Q1. React 的核心特点是什么？

**答：**

React 的核心特点通常可以概括为：声明式 UI、组件化开发、单向数据流，以及以状态驱动视图。

声明式 UI 的意思是，开发者不需要手动写大量 DOM 操作代码，而是描述“当前状态下界面应该是什么样子”，然后由 React 负责根据状态变化去更新真实界面。组件化则意味着我们把页面拆成多个职责清晰、可复用的组件单元，这样有助于复用、测试和维护。

单向数据流是 React 很重要的设计理念。父组件通过 props 把数据传给子组件，子组件通过回调或事件方式通知父组件更新。这样虽然有时会觉得比双向绑定麻烦一点，但优势是数据来源更清晰，问题定位更容易。

所以如果面试官问一句话总结，我一般会回答：React 是一个通过组件化和声明式方式、用状态驱动 UI 的前端库。

## Q2. React 和 Vue 的主要区别可以怎么理解？

**答：**

这题不建议答成“谁更好”，而应该从设计理念和工程习惯上做对比。

### 1. UI 表达方式不同

React 更偏向“JavaScript in UI”，通过 JSX 把视图和逻辑统一放在 JavaScript 语境里。  
Vue 更偏向“模板驱动”，模板语法对很多开发者来说更接近传统前端认知。

### 2. 框架定位不同

React 更像一个 UI 库，很多能力依赖生态组合，比如路由、状态管理、SSR 等。  
Vue 更强调渐进式和官方生态组合，很多中大型项目常见能力在官方方案里就能较完整覆盖。

### 3. 工程自由度不同

React 给开发者的自由度更高，但也意味着团队规范更重要。  
Vue 官方推荐路径更集中，落地往往更统一。

### 4. 心智差异

React 更强调函数式思维、不可变数据、状态驱动。  
Vue 则更强调响应式系统和模板层抽象。

所以更成熟的说法是：两者都能支撑大型项目，区别更多在于工程哲学、团队习惯和生态选择，而不是简单的强弱对比。

## Q3. JSX 是什么？它最终会变成什么？

**答：**

JSX 是 JavaScript 的语法扩展，允许我们在 JavaScript 中写类似 HTML 的结构。

```tsx
const element = <h1>Hello</h1>
```

很多人初学时会误以为 JSX 是模板语言，其实更准确地说，它是一种编译期语法糖。它最终会被 Babel 或其他编译器转成 React 元素创建调用，比如早期常见的是：

```tsx
React.createElement('h1', null, 'Hello')
```

在现代 JSX runtime 下，编译结果可能不是直接调用 `React.createElement`，但本质一样，都是生成描述 UI 的对象结构。

所以 JSX 的关键理解有三点：

- 它不是字符串模板
- 它不是浏览器原生语法
- 它最终会被编译成 React 元素描述

## Q4. React 组件和 React 元素有什么区别？

**答：**

React 元素和组件是两个不同层次的概念。

### React 元素

React 元素是对界面的最小描述单元，本质上是普通对象，里面记录了：

- 类型
- props
- children

可以把它理解成“UI 的描述结果”。

### React 组件

组件则是生成 React 元素的函数或类。现代 React 基本以函数组件为主。

```tsx
function Welcome() {
  return <h1>Hello</h1>
}
```

因此更准确的说法是：

- 组件是“描述 UI 生成逻辑”的单元
- 元素是“组件运行后产出的 UI 描述对象”

这个区别在理解渲染、组件复用、JSX 编译以及 Fiber 架构时都很重要。

## Q5. 什么是受控组件和非受控组件？

**答：**

这题一般围绕表单处理展开。

### 受控组件

受控组件指表单值完全由 React state 驱动。

```tsx
const [value, setValue] = useState('')
<input value={value} onChange={(e) => setValue(e.target.value)} />
```

它的优点是：

- 数据源统一
- 更适合实时校验
- 更适合联动和复杂表单逻辑

### 非受控组件

非受控组件是由 DOM 自己维护值，React 只在必要时通过 ref 读取。

```tsx
const inputRef = useRef<HTMLInputElement>(null)
```

优点是：

- 代码更简单
- 某些场景下性能负担更小

实践中我通常会说：

- 复杂业务表单优先受控组件
- 简单输入或第三方封装场景可考虑非受控

这题考察的是你有没有真正写过表单，而不只是背概念。

## Q6. `useState` 的特点是什么？为什么状态更新看起来不是立刻生效？

**答：**

`useState` 用于给函数组件添加状态。

```tsx
const [count, setCount] = useState(0)
```

它的核心特点是：状态更新不会像普通变量赋值那样“立刻同步改当前作用域里的值”，而是会触发 React 发起一次新的渲染调度。

也就是说，当你执行：

```tsx
setCount(count + 1)
```

本质上是在告诉 React：“下一轮渲染请使用新的状态”，而不是当前这一行往下走时 `count` 立刻变了。

为什么这么设计？因为 React 要批量协调更新、控制渲染节奏、保证组件函数是基于某个稳定快照执行的。

这题常见追问是“如果新值依赖旧值怎么办”，标准答案是用函数式更新：

```tsx
setCount((prev) => prev + 1)
```

因为这样能避免闭包读到旧值，也更符合 React 的调度模型。

## Q7. 为什么 React 强调不可变更新？

**答：**

React 强调不可变更新，不是为了形式，而是因为它和状态比较、性能优化、调试体验都有直接关系。

### 1. 更容易判断变化

很多场景下 React 或开发者自己的优化逻辑，会基于引用是否变化做浅比较。如果你直接修改原对象，引用没变，就容易导致更新判断不准确。

### 2. 更利于调试和追踪

不可变更新会让每次状态变化更清晰，尤其是结合时间旅行调试、日志记录、状态回溯时更明显。

### 3. 更契合函数式思维

React 函数组件和 Hook 的思维方式，本质上就更偏向不可变数据流。

错误写法：

```tsx
user.name = 'Jerry'
setUser(user)
```

正确写法：

```tsx
setUser((prev) => ({
  ...prev,
  name: 'Jerry',
}))
```

所以这题要答出的重点不是“React 不让改”，而是“不可变更新能让状态变化更可预测、更容易优化”。

## Q8. `useEffect` 是做什么的？

**答：**

`useEffect` 用于处理副作用。

这里的副作用指的是：不是单纯根据 props 和 state 推导 UI，而是需要和外部世界交互的逻辑，比如：

- 请求数据
- 订阅事件
- 操作 DOM
- 定时器
- 与浏览器 API 或第三方库同步

React 推荐的思路是：

- 渲染阶段保持尽量纯
- 副作用放到 effect 阶段处理

这题的关键点是：`useEffect` 不是“放所有逻辑的地方”。  
很多新手会把本该在 render 中计算的派生值、甚至本该写成事件回调的逻辑一股脑塞进 `useEffect`，这是不合理的。

更成熟的判断标准是：

- 如果只是根据已有状态算出新值，优先在渲染中算或用 `useMemo`
- 如果确实要和外部系统同步，再考虑 `useEffect`

## Q9. `useEffect` 的依赖数组分别代表什么？

**答：**

依赖数组的语义一定要讲清楚。

### 不传依赖数组

```tsx
useEffect(() => {})
```

每次渲染后都会执行。

### 传空数组 `[]`

```tsx
useEffect(() => {}, [])
```

通常表示只在首次挂载后执行一次。

### 传具体依赖 `[a, b]`

```tsx
useEffect(() => {}, [a, b])
```

首次执行后，只要 `a` 或 `b` 变化，就会重新执行。

但这题真正考的不是背规则，而是：你是否理解 effect 中引用到的响应值，应尽量真实地体现在依赖数组中。  
如果依赖写少了，容易出现旧值问题；写错了，又可能造成不必要重复执行甚至死循环。

## Q10. `useEffect` 为什么需要清理函数？

**答：**

因为很多副作用不是“执行一次就完了”，而是会建立持续性的外部连接或资源占用，比如：

- 定时器
- 事件监听
- WebSocket
- 订阅
- 手动创建的实例

React 提供清理函数，就是为了在：

- 组件卸载时
- effect 因依赖变化重新执行前

把旧副作用清掉，避免资源泄漏和逻辑重复。

```tsx
useEffect(() => {
  const timer = setInterval(() => {}, 1000)
  return () => clearInterval(timer)
}, [])
```

这题如果能顺带提到“严格模式开发环境下 effect 可能被额外检查执行，从而更容易暴露未清理副作用问题”，回答会更完整。

## Q11. `useMemo` 和 `useCallback` 有什么区别？

**答：**

很多人背成“一个缓存值，一个缓存函数”，这没错，但还不够。

### `useMemo`

用于缓存某个计算结果。

```tsx
const total = useMemo(() => calc(list), [list])
```

适合：

- 昂贵计算
- 需要稳定对象引用

### `useCallback`

用于缓存函数引用。

```tsx
const onSubmit = useCallback(() => {}, [])
```

适合：

- 函数作为 props 传给被 `memo` 包裹的子组件
- 函数本身是其他 Hook 的依赖

### 更深入的理解

`useCallback(fn, deps)` 本质上可以看作 `useMemo(() => fn, deps)` 的语义化版本。

同时还要强调：两者都不是“默认就该加”。如果组件本身不慢、没有引用稳定性要求，滥用这些 Hook 只会增加复杂度。

## Q12. `useRef` 有哪些常见用途？

**答：**

`useRef` 有两个非常核心的用途。

### 1. 获取 DOM 引用

这是最常见的场景，比如：

- 输入框自动聚焦
- 读取元素尺寸
- 调用 DOM API

### 2. 保存一个不会触发重渲染的可变值

这非常关键，很多人只记得它是“拿 DOM 的”。

例如：

- 保存定时器 id
- 保存上一次值
- 保存最新回调引用
- 保存第三方实例

它和 `useState` 的关键区别是：修改 `ref.current` 不会触发重新渲染。

所以如果一个值需要驱动 UI，就应该放 `state`；如果只是需要跨渲染保留，但不需要触发 UI 更新，就适合用 `ref`。

## Q13. 什么是闭包陷阱？在 React 中常见在哪些地方？

**答：**

闭包陷阱指的是：某个异步回调拿到的是“某次渲染时的旧变量”，而不是当前最新状态。

这是 React 函数组件里非常高频的问题，因为每次渲染都会形成新的作用域快照。

常见场景：

- `setInterval`
- `setTimeout`
- effect 中注册的事件回调
- 某些延迟执行函数

典型例子是：

```tsx
useEffect(() => {
  const timer = setInterval(() => {
    console.log(count)
  }, 1000)
  return () => clearInterval(timer)
}, [])
```

如果依赖数组是空的，回调里拿到的 `count` 可能一直是初始值。

常见解决方案：

- 正确补依赖
- 使用函数式更新
- 用 ref 保存最新值
- 重新设计副作用结构

这题其实是在考你对“函数组件每次渲染是独立快照”这个核心认知是否清楚。

## Q14. `key` 的作用是什么？为什么不推荐用 index？

**答：**

`key` 是 React 在协调列表节点时识别元素身份的依据。

它不是给你看的，而是给 React 用来判断：

- 哪个节点是新增的
- 哪个节点是复用的
- 哪个节点要移动
- 哪个节点要销毁

如果 key 不稳定，比如直接用数组 index，那么一旦中间插入、删除、重排，就容易导致：

- 输入框内容错位
- 组件内部状态复用到错误项
- 动画和过渡异常

因此更成熟的结论是：

- 静态不会重排的简单列表，用 index 有时问题不大
- 但动态业务列表应尽量使用稳定唯一 id

## Q15. React 组件通信有哪些方式？

**答：**

React 组件通信要按层级回答。

### 1. 父传子：props

这是最基础的方式。

### 2. 子传父：回调函数

父组件把函数传给子组件，子组件触发时把数据传回来。

### 3. 跨层级：Context

用于避免 props drilling。

### 4. 更大范围共享：状态管理库

例如：

- Redux Toolkit
- Zustand
- Jotai

### 5. 父组件调用子组件能力

可以结合 `forwardRef`、`useImperativeHandle` 等方式暴露能力。

更成熟的回答会强调：React 本身仍然优先倡导单向数据流，Context 和状态管理是对特殊通信场景的补充，而不是替代 props 设计。

## Q16. `props` 和 `state` 的区别是什么？

**答：**

这题表面很基础，但其实在考组件边界认知。

### props

- 来自父组件
- 是组件的外部输入
- 从子组件视角看通常应视为只读

### state

- 由组件自身维护
- 代表组件内部会变化的数据
- 更新会触发重新渲染

更本质地说：

- props 描述“外部世界给我的条件”
- state 描述“我自己的内部状态”

如果一个数据既可能被父组件控制，又想在组件内部修改，就要认真设计控制模式，否则容易出现受控/非受控混乱。

## Q17. 什么是高阶组件 HOC？什么是 Render Props？

**答：**

这两个是 Hooks 之前 React 中非常重要的逻辑复用模式。

### 高阶组件 HOC

HOC 是“接收组件，返回增强后组件”的函数。

它适合做：

- 权限包装
- 埋点包装
- 通用逻辑增强

### Render Props

通过把一个函数作为 prop 传入，让子组件把内部状态或逻辑暴露出来，由父组件决定怎么渲染。

这两种模式今天仍然有价值，但在很多场景下已经被自定义 Hook 替代了。  
面试中如果能答出“它们是 React 逻辑复用演进历史的一部分，而 Hook 是更自然的现代方案”，会更显得理解全面。

---

## Q18. 自定义 Hook 有什么意义？

**答：**

自定义 Hook 的核心意义是复用“状态逻辑”，而不是复用 UI。

比如：

- `useRequest`
- `useDebounce`
- `useWindowSize`
- `useIntersection`

它的优点是：

- 逻辑复用更自然
- 来源更明确
- 避免 HOC 嵌套地狱
- 避免 render props 过度嵌套

同时要强调一点：自定义 Hook 只是把逻辑提炼出来，不会凭空创建独立状态池。每个组件调用同一个 Hook，本质上还是各自拥有一份状态实例。

这题如果能答到这层，通常就不是停留在“会用 Hook”的层面了。

## Q19. `memo`、`useMemo`、`useCallback` 分别适合什么场景？

**答：**

这题一定要区分对象。

### `memo`

用于缓存组件本身。  
当 props 没变化时，组件可以跳过重新渲染。

### `useMemo`

用于缓存某个计算结果。

### `useCallback`

用于缓存函数引用。

它们之间的关系往往是配合使用，而不是替代关系。  
例如父组件用 `useCallback` 保持回调引用稳定，子组件再用 `memo` 避免因为 props 引用变化而重复渲染。

但最后一定要补一句：优化前先确认瓶颈，不要把这些 Hook 当默认模板，否则可能得不偿失。

## Q20. 什么是 Fiber？为什么 React 需要 Fiber？

**答：**

Fiber 可以理解成 React 新的协调架构，它把原本可能很大的渲染任务拆成更小的工作单元，以支持更灵活的调度。

为什么需要它？因为旧的同步递归渲染模式一旦开始，就很难中断。如果组件树很大，渲染过程可能长时间占用主线程，导致用户输入、动画、滚动等交互卡顿。

Fiber 的重要意义在于：

- 可以拆分任务
- 可以中断和恢复
- 可以根据优先级安排更新
- 为并发能力打基础

所以 Fiber 不是一个“新 API”，而是 React 内部为现代交互复杂度做出的架构升级。

## Q21. React 18 有哪些重要变化？

**答：**

React 18 的常见高频点包括：

- `createRoot`
- 自动批处理增强
- 并发能力基础
- `startTransition`
- `useTransition`
- `useDeferredValue`

React 18 在更多场景下会自动合并多次状态更新，减少无意义重渲染。

### `startTransition`

适合把“非紧急更新”标记成低优先级，例如：

- 搜索结果过滤
- 大列表切换
- 非关键 UI 切换

面试中如果能把 React 18 概括为“更强的调度能力和更好的交互体验基础设施”，会比单纯背 API 更好。

## Q22. 什么是 Context？适合什么场景？

**答：**

Context 用于跨层级共享数据，避免中间层只是为了传值而不断透传 props。

典型场景：

- 主题
- 当前用户
- 语言环境
- 表单上下文

但它不是“全局状态银弹”。  
因为当 Context value 变化时，相关消费组件都会重新参与更新，所以如果把频繁变化的大型业务状态一股脑塞进去，反而会带来额外复杂度和性能问题。

成熟的判断是：

- 轻量全局上下文：Context
- 复杂全局业务状态：专门状态管理方案

## Q24. React 性能优化有哪些常见手段？

**答：**

这题建议按思路回答，而不是只堆 API。

### 1. 降低不必要渲染

- 合理拆分组件
- 控制状态粒度
- 避免无意义状态提升

### 2. 利用缓存机制

- `memo`
- `useMemo`
- `useCallback`

### 3. 大列表优化

- 虚拟列表
- 分页
- 延迟渲染

### 4. 加载优化

- 路由懒加载
- 动态导入
- 图片优化

### 5. 运行时优化

- 避免主线程长任务
- 避免渲染阶段高成本计算

最后最好补一句：真正的性能优化应建立在定位瓶颈基础上，而不是先写一堆 `memo`。

## Q25. 你如何理解 React 在中大型项目中的优势与边界？

**答：**

我通常会从“优势”和“边界”两部分回答。

### 优势

- 组件模型成熟
- 生态丰富
- 与 TypeScript、SSR、跨端方案配合成熟
- 对复杂交互和大型前端工程支持很好
- 函数组件和 Hook 让逻辑复用更灵活

### 边界

- 自由度高也意味着规范要求更高
- 如果团队没有统一的状态边界、组件设计和工程规范，很容易写散
- React 本身不是完整工程方案，很多能力要靠生态与团队约定落地

所以更成熟的结论是：React 很适合复杂前端系统，但它的上限很大程度取决于团队工程能力，而不只是框架本身。
