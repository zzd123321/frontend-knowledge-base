# 前端性能优化核心知识点

前端性能优化不是背几个技巧名词，而是围绕“用户感觉快不快”做系统化设计。真正有效的性能工作，一定包含四件事：

- 明确目标
- 采集数据
- 定位瓶颈
- 验证收益

如果只会说“加缓存、做懒加载、压缩资源”，但说不清为什么慢、慢在哪、优化后改善了哪个指标，这种优化很难真正落地。

---

## 1. 性能优化到底在优化什么

用户并不关心你的 bundle 是多少 KB，他们只关心：

- 页面打开得够不够快
- 点击后有没有立刻响应
- 页面滚动顺不顺
- 界面会不会突然乱跳
- 弱网时能不能正常用

所以性能优化的本质是提升用户体验，而不是机械追求某个数字。

常见目标：

- 更快的首屏
- 更短的可交互时间
- 更流畅的交互和动画
- 更稳定的布局
- 更低的资源消耗

---

## 2. 性能优化的主要维度

前端性能通常可以拆成这些维度：

- 加载性能
- 渲染性能
- 运行时性能
- 网络性能
- 资源体积
- 构建性能
- 监控与治理

很多问题其实会跨维度。

例如“页面卡”可能来自：

- JS 执行太重
- 请求瀑布太长
- 图片太大
- 重渲染太多
- 主线程被长任务阻塞

所以性能分析不能只盯一个点。

---

## 3. 性能指标为什么重要

优化一定要和指标绑定，否则就很容易陷入“感觉变快了”的主观判断。

常见作用：

- 量化体验
- 识别瓶颈
- 比较优化前后收益
- 建立团队统一目标

指标不是目的，但没有指标几乎不可能持续做好性能。

---

## 4. Core Web Vitals

当前前端性能里最重要的一组指标是 Core Web Vitals。

### LCP

Largest Contentful Paint，最大内容绘制。

关注点：

- 主要内容什么时候出现在屏幕上

它通常反映首屏主要内容是否够快展示。

### INP

Interaction to Next Paint，交互到下一次绘制。

关注点：

- 用户操作后，界面多久能真正反馈

它反映交互响应性。

### CLS

Cumulative Layout Shift，累计布局偏移。

关注点：

- 页面会不会突然乱跳

它反映视觉稳定性。

---

## 5. 除了 Web Vitals 还要看什么

实际工程里，还经常看：

- FCP：首次内容绘制
- TTFB：首字节时间
- TBT：总阻塞时间
- Speed Index
- 首屏请求数
- JS 执行时长
- 长任务数量
- 错误率和超时率

这些指标能帮助你更完整地理解问题链路。

例如：

- TTFB 高，可能偏后端或网络
- FCP 慢，可能偏资源阻塞
- INP 差，可能偏主线程阻塞
- CLS 高，可能偏布局不稳定

---

## 6. 关键渲染路径

浏览器从拿到 HTML 到页面显示，大致会经历：

1. 解析 HTML，构建 DOM
2. 解析 CSS，构建 CSSOM
3. 合成 Render Tree
4. Layout
5. Paint
6. Composite

这条链路越长、阻塞越多，首屏越慢。

所以前端常见优化很多都围绕这条路径展开：

- 更早拿到 HTML
- 更早拿到关键 CSS
- 减少阻塞脚本
- 降低首屏 JS 成本
- 推迟非关键资源加载

---

## 7. 导致首屏慢的常见原因

首屏慢通常不是一个原因，而是几个问题叠加：

- HTML 返回慢
- DNS / TCP / TLS 建连慢
- 首屏资源请求太多
- JS 包太大
- 主线程初始化太重
- 图片太大
- 字体阻塞
- 接口请求瀑布太深

优化首屏，关键不是“把所有东西都压一遍”，而是先判断瓶颈在哪一段。

---

## 8. 首屏加载优化

常见手段：

- 减少首屏 JS
- 路由级和组件级懒加载
- 提取关键 CSS
- 优化首屏图片
- 预加载关键资源
- SSR / SSG
- 使用 CDN 和缓存

要特别注意：

- 首屏优化核心是“优先级排序”
- 当前不需要的东西，不要和首屏抢资源

---

## 9. 资源优先级意识

并不是所有资源都应该一起加载。

更合理的思路是：

- 当前必须用的先加载
- 很快会用的预加载
- 未来可能会用的预取
- 暂时不用的延后

常见资源提示：

- `preload`
- `prefetch`
- `preconnect`
- `dns-prefetch`

它们解决的问题不同，不能混用。

---

## 10. 图片优化

图片通常是页面体积大户。

常见优化方向：

- 选择合适格式
- 控制实际展示尺寸
- 压缩质量
- 响应式图片
- 懒加载
- CDN 裁剪和转码

### 格式选择

- `jpg`：适合照片类
- `png`：适合透明图和高保真简单图
- `webp`：兼顾体积和质量
- `avif`：更强压缩率，但要关注兼容和处理链路

### 响应式图片

不要用一张超大图在移动端强行缩小显示。

应该根据场景输出不同尺寸。

---

## 11. 图片懒加载

对于非首屏图片，常见做法是懒加载。

好处：

- 减少首屏请求数
- 降低首屏带宽竞争
- 缩短首屏关键资源完成时间

常见实现：

- 原生 `loading="lazy"`
- `IntersectionObserver`

但要注意：

- 首屏关键图不要错误懒加载
- LCP 图通常应该被优先加载

---

## 12. 字体优化

字体很容易被忽略，但它对首屏和稳定性影响不小。

常见优化：

- 只加载必要字重
- 做字体子集化
- 使用 `font-display: swap`
- 预加载关键字体
- 优先系统字体或降级方案

字体问题常见表现：

- 白屏等字
- 文本闪烁
- 首屏迟迟不稳定

---

## 13. CSS 对性能的影响

CSS 不只是样式问题，也会影响首屏。

关键点：

- CSS 会影响渲染树构建
- 关键 CSS 获取慢，会拖慢首屏展示
- 大量无效样式会增加解析和匹配成本

优化方向：

- 控制样式体积
- 清理无用样式
- 提取关键 CSS
- 避免选择器过度复杂

---

## 14. JavaScript 体积优化

JS 往往同时影响：

- 下载时间
- 解析时间
- 编译时间
- 执行时间

优化方向：

- Tree Shaking
- 动态导入
- 按需加载
- 替换重型依赖
- 控制 polyfill 体积
- 减少首屏无关逻辑

一句话：

- 少发一点 JS，常常比“优化执行细节”更划算

---

## 15. 代码分割

代码分割的核心不是“拆得越碎越好”，而是：

- 让当前路径只加载当前需要的代码

常见拆分方式：

- 路由级拆分
- 组件级懒加载
- 第三方库独立拆包
- 大功能模块按需加载

做得不好也会出问题：

- 拆包过碎，网络请求太多
- 重复依赖进入多个 chunk
- 首屏仍然加载太多同步入口依赖

---

## 16. 网络性能优化

网络层常见优化：

- 使用 CDN
- 开启 Gzip / Brotli
- 合理使用 HTTP/2 或 HTTP/3
- 减少请求数量
- 避免请求瀑布
- 做缓存
- 就近接入和边缘加速

前端要建立一个意识：

- 不是每个请求都慢，而是某些关键请求慢会拖垮整条链路

---

## 17. 缓存优化

缓存是最划算的性能手段之一。

常见缓存层次：

- 浏览器缓存
- CDN 缓存
- Service Worker 缓存
- 应用内存缓存

典型策略：

- 带 hash 的静态资源长缓存
- HTML 短缓存或协商缓存
- 业务接口按实时性决定是否缓存

如果缓存策略设计得好，很多“性能优化”根本不用每次重新做。

---

## 18. 压缩与传输优化

常见内容压缩：

- Gzip
- Brotli

一般来说：

- Brotli 压缩率更高
- 但生成成本通常也更高

传输优化的价值在于：

- 相同内容更少字节
- 弱网收益更明显

但要注意：

- 压缩对已经很小或本身已压缩的资源未必收益大

---

## 19. 请求瀑布是什么

请求瀑布指的是：

- 后一个请求依赖前一个请求返回，导致串行等待

比如：

1. 先请求用户信息
2. 再根据用户信息请求菜单
3. 再根据菜单请求权限
4. 再请求页面数据

这种链路很容易拖慢首屏和交互。

优化思路：

- 能并发就并发
- 提前拿公共上下文
- 后端聚合接口
- 避免无意义的串行依赖

---

## 20. 主线程为什么重要

浏览器主线程要同时负责很多事情：

- 执行 JS
- 样式计算
- 布局
- 绘制
- 处理用户输入

如果主线程被长任务占满，用户点击虽然发生了，但界面可能迟迟不给反馈。

这也是为什么现代性能优化里，运行时优化和交互响应优化越来越重要。

---

## 21. 长任务与阻塞

长任务通常指主线程上执行时间过长的任务。

常见来源：

- 大循环
- 大量 JSON 解析
- 重计算
- 首次渲染初始化逻辑过重
- 大量同步 DOM 操作

它会导致：

- 点击无响应
- 输入卡顿
- 滚动不流畅
- INP 变差

---

## 22. 如何拆长任务

常见思路：

- 切分任务
- 分帧处理
- 延后非关键逻辑
- 使用 `requestIdleCallback`
- 使用 Worker
- 减少同步计算

例如：

- 首屏不需要的统计逻辑延后
- 大型计算放进 Worker
- 大列表分批渲染

核心目标是：

- 不让主线程一次性背太多工作

---

## 23. 回流、重绘与合成

### 回流

布局重新计算，通常代价更高。

### 重绘

只重新绘制，不重新布局。

### 合成

把图层合成到屏幕上显示。

性能优化里常见原则：

- 尽量少触发布局
- 动画尽量使用更接近合成层的属性

---

## 24. 布局抖动

布局抖动常见于：

- 先读布局信息
- 再写样式
- 再读布局
- 再写样式

例如频繁调用：

- `offsetWidth`
- `clientHeight`
- `getBoundingClientRect()`

然后紧接着改样式，会导致浏览器频繁强制同步布局。

优化思路：

- 批量读
- 批量写
- 避免读写交错

---

## 25. 动画优化

更适合做高频动画的属性通常是：

- `transform`
- `opacity`

原因是它们通常不会频繁触发布局。

不太适合高频动画的属性常见有：

- `width`
- `height`
- `top`
- `left`

动画优化核心不是“让动画更花”，而是“让动画更稳”。

---

## 26. 高频事件优化

常见高频事件：

- `scroll`
- `resize`
- `mousemove`
- 输入联想

常见优化手段：

- 防抖
- 节流
- `requestAnimationFrame`
- 被动事件监听

### 防抖

适合：

- 搜索输入
- resize 结束后计算

### 节流

适合：

- 滚动监听
- 鼠标移动

---

## 27. 列表渲染优化

大列表很容易成为卡顿源头。

常见问题：

- DOM 节点太多
- 每个节点逻辑太重
- 滚动时频繁重排

常见优化：

- 虚拟列表
- 分页
- 无限滚动
- 分段渲染
- 减少列表项内复杂计算

虚拟列表的核心思想是：

- 只渲染视口附近需要看到的内容

---

## 28. 框架层性能优化

框架会帮你做很多事，但也可能带来额外开销。

框架层常见优化方向：

- 减少无意义重渲染
- 控制状态粒度
- 避免大范围联动
- 减少首屏组件树复杂度
- 按需加载大型组件

优化前一定先确认：

- 真正的问题是框架渲染，还是请求、逻辑、样式、资源

---

## 29. React 常见性能优化思路

- 状态尽量就近管理
- 减少父组件无意义更新传递
- 识别昂贵组件
- 适度使用 `memo`
- 让列表有稳定 key
- 避免在渲染中做大计算
- 用懒加载拆分页面和重组件

注意：

- 不要把 `memo`、`useMemo`、`useCallback` 当银弹
- 没有性能问题时过度优化，反而会增加复杂度

---

## 30. Vue 常见性能优化思路

- 合理拆分组件
- 避免深层响应式数据被大范围依赖
- 使用 `computed` 缓存派生结果
- 对稳定内容使用 `v-once`
- 大列表配合虚拟化
- 避免模板中复杂表达式和大计算

Vue 的响应式机制很强，但如果依赖关系设计混乱，也会带来多余更新。

---

## 31. Worker 与多线程

Web Worker 适合处理：

- 大量计算
- 大数据处理
- 编码解码
- 大型文本解析

它的价值是：

- 把重计算移出主线程

但要注意：

- Worker 不能直接操作 DOM
- 线程间通信有序列化成本

所以不是所有逻辑都适合丢给 Worker。

---

## 32. 内存性能

前端性能不只有速度，还有资源占用。

常见内存问题：

- 事件监听未清理
- 定时器未清理
- 闭包长期持有大对象
- 大量缓存不释放
- DOM 节点被意外引用

内存问题的典型表现：

- 页面越用越卡
- 长时间运行后崩溃
- 标签页占用持续上涨

---

## 33. 内存泄漏排查思路

可以重点关注：

- 组件卸载后对象是否仍被引用
- 大数组、大 Map 是否持续增长
- observer、listener、timer 是否销毁
- 截图前后 heap 是否持续上涨

常用工具：

- Chrome DevTools Memory
- Heap Snapshot
- Performance 面板

---

## 34. SSR、SSG、CSR 的性能差异

### CSR

客户端渲染开发灵活，但首屏通常更依赖 JS 下载、执行和接口完成。

### SSR

服务端先输出 HTML，首屏更友好，也更利于 SEO。

### SSG

构建阶段生成静态页面，适合内容稳定页面，首屏通常更快。

选择哪种方式，不能只看“谁快”，还要看：

- 页面类型
- 数据实时性
- 部署复杂度
- SEO 需求

---

## 35. Service Worker 与离线缓存

Service Worker 能做的事包括：

- 缓存资源
- 拦截请求
- 支持离线访问
- 实现更复杂的缓存策略

适合场景：

- PWA
- 弱网容错
- 离线可读内容

但要注意：

- 缓存策略设计不当也会导致更新不及时、调试困难

---

## 36. 构建性能

性能不只影响用户，也影响开发效率。

构建性能常见关注点：

- 冷启动速度
- 热更新速度
- 打包时长
- 类型检查耗时
- 测试运行耗时

优化方向：

- 升级工具链
- 缓存结果
- 缩小处理范围
- 减少不必要插件
- 分离类型检查和转译

开发体验差，团队迭代速度也会慢。

---

## 37. 包体积分析

如果不知道大包来自哪里，就很难优化。

常见分析思路：

- 看主包大小
- 看第三方依赖占比
- 看重复依赖
- 看是否把只在少量场景使用的模块放进首屏

典型优化：

- 替换重型库
- 局部引入
- 延迟加载
- 去掉无用 polyfill

---

## 38. 性能监控

没有监控，优化就很难长期持续。

常见监控内容：

- Web Vitals
- 页面加载耗时
- 资源加载耗时
- 长任务
- JS 错误
- 接口失败率
- 页面白屏或卡顿

常用工具：

- Lighthouse
- Chrome DevTools
- Performance API
- Web Vitals SDK
- 前端监控平台

---

## 39. 实验室数据与真实用户数据

性能分析通常有两类数据：

- Lab Data：实验室环境数据
- RUM：真实用户监控数据

实验室数据适合：

- 稳定复现
- 对比优化方案

真实用户数据适合：

- 观察实际环境表现
- 看不同设备、地区、网络下的体验差异

真正成熟的性能治理，一般两者都要看。

---

## 40. 性能优化的方法论

推荐流程：

1. 明确问题场景
2. 选择指标
3. 采集和复现
4. 拆解瓶颈
5. 设计优化方案
6. 验证收益
7. 建立持续监控

这个流程很重要，因为很多“优化”之所以无效，就是因为一开始没有准确定义问题。

---

## 41. 常见性能误区

- 一上来就优化细枝末节
- 没有测量就开始改
- 盲目追求极限压缩
- 所有页面统一使用同一套策略
- 把复杂度极高的优化强塞给没有收益的场景
- 只看首屏，不看交互和稳定性

性能优化不是“越多技巧越好”，而是“用最合适的方法解决最真实的问题”。

---

## 42. 前端性能最值得掌握的主线

如果只抓一条主线，就抓这条：

1. 用户为什么会觉得慢
2. 慢发生在加载、渲染、交互还是内存层
3. 用指标定位问题
4. 用合适手段解决问题
5. 用监控验证和持续治理
