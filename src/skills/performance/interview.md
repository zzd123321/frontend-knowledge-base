# 前端性能优化面试题

## Q1. 前端性能优化通常从哪些方面入手？

**答：**

这题最好不要背成一串零碎技巧，而要给出体系化框架。

我通常会从以下几个维度回答：

### 1. 加载性能

解决“页面打开得快不快”的问题，比如：

- 首屏资源体积
- 路由懒加载
- 图片优化
- 缓存
- CDN

### 2. 渲染性能

解决“界面显示和滚动流不流畅”的问题，比如：

- 回流重绘
- 大列表渲染
- 动画卡顿
- 布局抖动

### 3. 运行时性能

解决“交互卡不卡”的问题，比如：

- 主线程长任务
- 大量同步计算
- 不必要的框架重渲染

### 4. 网络性能

解决“请求慢不慢、请求是否合理”的问题，比如：

- 请求数量
- 缓存命中率
- 接口瀑布流
- 资源压缩

### 5. 工程与监控

解决“能否持续优化”的问题，比如：

- 构建体积分析
- Web Vitals 监控
- Lighthouse
- RUM

所以成熟回答的关键不是列 API，而是先给出性能优化地图。

## Q2. Web Vitals 主要指标有哪些？

**答：**

当前最值得关注的核心 Web Vitals 通常包括：

- LCP
- INP
- CLS

### LCP

Largest Contentful Paint，最大内容绘制。  
它衡量用户何时能看到页面主要内容，通常反映首屏加载体验。

### INP

Interaction to Next Paint。  
它衡量用户交互之后，到界面真正反馈出来之间的延迟，反映交互流畅度。

### CLS

Cumulative Layout Shift。  
它衡量页面视觉稳定性，也就是界面是否会突然乱跳。

如果面试官继续追问，我会补一句：过去很多人习惯看 FID，但现在更推荐关注 INP，因为它更能代表真实交互表现。

## Q3. 什么是关键渲染路径？

**答：**

关键渲染路径指的是浏览器把 HTML、CSS、JS 转换成用户可见页面的那条核心处理链路。

简化流程通常是：

1. 解析 HTML，构建 DOM
2. 解析 CSS，构建 CSSOM
3. 生成渲染树
4. Layout
5. Paint
6. Composite

为什么它重要？因为首屏优化很多时候本质上就是在优化这条路径中的阻塞点，比如：

- CSS 是否阻塞渲染
- JS 是否阻塞解析
- 首屏图片是否过大
- 关键资源是否过多

如果能把性能问题落回到关键渲染路径上去解释，说明你理解的不是孤立技巧，而是浏览器运行机制。

## Q4. 如何优化首屏加载速度？

**答：**

首屏优化是前端性能里最常见的考点。  
我一般会从“减少关键资源体积”和“优化关键路径顺序”两方面来讲。

### 1. 减少首屏 JS

- 做代码分割
- 路由懒加载
- 大模块异步加载
- 避免整库引入

### 2. 优化首屏静态资源

- 压缩图片
- 使用现代图片格式
- 减少首屏字体和大图资源
- 内联关键 CSS 或保证关键 CSS 尽早到达

### 3. 利用缓存与 CDN

- 静态资源使用 hash + 强缓存
- CDN 加速静态资源分发

### 4. 服务端方案

- SSR
- SSG

特别适合内容型页面、SEO 敏感页面和首屏要求高的页面。

### 5. 避免请求瀑布

减少“必须等上一个请求结束，才能发下一个请求”的情况。

所以更好的回答方式不是只说“懒加载、压缩”，而是把首屏优化解释成：减少关键路径中的阻塞体积和依赖链条。

---

## Q5. 为什么 JS 包太大不仅影响下载，还影响性能？

**答：**

因为一个 JavaScript 包的成本不只是“下载”。

浏览器拿到 JS 之后，还要做：

- 解析
- 编译
- 执行

这些工作都发生在主线程上。  
所以即使网络还不错，大包依然会导致：

- 首屏可交互变慢
- 主线程被占用
- 用户点击和输入卡顿

这也是为什么前端性能优化里要特别重视：

- 减少首屏 JS
- 避免无意义依赖
- 按需加载

所以“大包问题”本质上既是网络问题，也是主线程执行问题。

## Q6. 回流和重绘有什么区别？

**答：**

### 回流

也叫 Reflow 或 Layout。  
当元素的几何信息需要重新计算时会触发回流，比如：

- 宽高变化
- 位置变化
- 字体大小变化
- DOM 结构变化

### 重绘

当元素外观变化但不影响布局时会触发重绘，比如：

- 颜色变化
- 背景变化

通常回流开销更大，因为它会影响布局计算，甚至可能影响整片区域。

面试时更成熟的回答是：

- 回流一定会引发重绘
- 重绘不一定引发回流

## Q7. 为什么动画优先用 `transform` 和 `opacity`？

**答：**

因为这两个属性通常不会频繁触发布局计算，更容易直接在合成阶段完成更新。

相比之下，如果动画频繁改：

- `top`
- `left`
- `width`
- `height`

就更容易引发布局和重绘，性能成本更高。

所以高频动画更推荐：

- `transform`
- `opacity`

这题真正考的是你是否理解“不是所有 CSS 属性的更新成本都一样”。

---

## Q8. 什么是防抖和节流？分别适合什么场景？

**答：**

### 防抖

事件频繁触发时，只在停止触发一段时间后执行一次。

适合：

- 搜索输入
- resize
- 表单实时校验

### 节流

在固定时间窗口内，最多只执行一次。

适合：

- scroll
- mousemove
- 高频拖拽

更准确地说：

- 防抖关注“最后一次”
- 节流关注“执行频率上限”

## Q9. 为什么大列表要考虑虚拟列表？

**答：**

因为 DOM 节点不是免费的。  
如果一次渲染成千上万个节点，会带来：

- 初始渲染慢
- 内存占用高
- 滚动卡顿
- diff 成本上升

虚拟列表的核心思想是：只渲染可视区域及其附近的少量节点，随着滚动动态替换内容，而不是把所有数据都真正挂到 DOM 上。

所以它本质上是在用“少量真实 DOM + 大量逻辑映射”换取性能。

## Q10. 图片优化有哪些常见方式？

**答：**

图片优化是前端最常见、也最实用的优化点之一。

常见方式包括：

- 压缩图片体积
- 选择更合适的格式，如 `webp`、`avif`
- 根据场景控制尺寸
- 使用响应式图片
- 做懒加载
- 用 CDN 分发

面试中更加分的点是说明：图片优化的目标不是“所有图都最小”，而是“在可接受画质下尽量减少带宽和渲染压力”。

---

## Q11. 浏览器缓存分哪两类？

**答：**

浏览器缓存主要分为：

- 强缓存
- 协商缓存

### 强缓存

命中后浏览器直接使用本地缓存，不发请求。

### 协商缓存

浏览器会向服务器确认资源是否变更，未变更则返回 `304`。

面试里最好顺带说一下：

- 强缓存通常靠 `Cache-Control` / `Expires`
- 协商缓存通常靠 `ETag` / `Last-Modified`

---

## Q12. `no-cache` 和 `no-store` 有什么区别？

**答：**

这是缓存语义里的高频细节题。

- `no-cache`：资源可以缓存，但每次使用前都必须先向服务器验证
- `no-store`：资源不允许缓存

很多人看到 `no-cache` 会误以为“不能缓存”，其实不是。  
因此如果这题答准，往往能体现你对缓存不是停留在字面理解。

## Q13. 为什么会出现布局偏移 CLS？怎么优化？

**答：**

CLS 反映的是页面视觉稳定性。  
用户正在看页面时，如果元素突然跳动，就会导致布局偏移。

常见原因包括：

- 图片没有预留宽高
- 广告位或异步内容突然插入
- 字体加载前后尺寸差异明显
- 顶部插入通知条把内容整体往下推

优化方式通常是：

- 给图片和视频预留尺寸
- 对动态模块预留占位空间
- 控制字体替换策略
- 避免在可见内容上方动态插入未知高度内容

这题的关键是说明：CLS 不是“页面抖一下”这么简单，而是直接影响用户操作准确性和观感。

## Q14. 什么是长任务？为什么它会影响体验？

**答：**

长任务指的是主线程被某段任务长时间占用，无法及时响应用户输入和渲染更新。

它会带来的问题包括：

- 点击没反应
- 输入延迟
- 动画掉帧
- 滚动卡顿

长任务的来源常见有：

- 大量同步计算
- 大 JSON 解析
- 渲染阶段做重计算
- 大量同步 DOM 操作

所以性能优化不只是“加载资源”，还包括控制主线程不要被长时间霸占。

## Q15. 如何减少不必要的框架重渲染？

**答：**

这题要结合框架机制来答，而不是只说 `memo`。

常见思路包括：

### 1. 合理拆分组件

把更新频繁和更新不频繁的区域拆开，缩小影响范围。

### 2. 控制状态粒度

不要把本来局部的状态无意义提升到很高层。

### 3. 缓存与稳定引用

在合适场景使用：

- React 的 `memo`、`useMemo`、`useCallback`
- Vue 的 `computed`、组件拆分等

### 4. 避免不必要的派生状态重复保存

如果一个值能由已有状态直接算出来，就不要额外再存一份 state。

所以更本质的说法是：减少重渲染不是靠“加缓存咒语”，而是靠更合理的状态边界和组件边界。

## Q16. SSR 和 CSR 在性能上有什么区别？

**答：**

### CSR

客户端渲染，浏览器先下载 JS，再由前端框架生成页面。

优点：

- 前后端分离清晰
- 开发方式灵活

缺点：

- 首屏通常不如 SSR
- SEO 支持较弱

### SSR

服务端先输出 HTML，再交给客户端接管。

优点：

- 首屏更快
- SEO 更好

缺点：

- 工程复杂度更高
- 服务端成本更高

所以这题的重点不是说谁一定更好，而是说明不同渲染模式优化的是不同阶段的用户体验。

## Q17. 如何分析前端性能瓶颈？

**答：**

成熟的排查方式通常是“先度量，再定位”。

常用工具包括：

- Chrome DevTools Network
- Chrome DevTools Performance
- Lighthouse
- Web Vitals 监控
- 前端埋点平台

分析时一般会看：

- 是网络慢，还是主线程慢
- 是资源体积大，还是请求链路长
- 是渲染卡，还是交互卡
- 是首屏慢，还是操作慢

这题如果只答“用 Lighthouse”，通常偏浅。  
真正重要的是你有没有把性能问题拆成网络、渲染、主线程、资源几个维度去定位。

---

## Q18. 为什么性能优化不能只看 Lighthouse 分数？

**答：**

因为 Lighthouse 更偏实验室环境评估，它反映的是某种模拟条件下的表现，而不是所有真实用户的实际体验。

现实中用户会有：

- 不同网络环境
- 不同设备性能
- 不同页面停留和交互方式

所以更成熟的性能体系应该同时关注：

- 实验室数据：Lighthouse、DevTools
- 真实用户数据：RUM、Web Vitals 埋点

一句话总结：Lighthouse 很有用，但它不是全部真相。

---

## Q19. 前端包体积优化有哪些常见方式？

**答：**

常见方式包括：

- Tree Shaking
- 动态导入
- 路由懒加载
- 按需加载组件库
- 替换重型依赖
- 做 bundle 分析

但更关键的是不要只停留在“怎么减”，而要知道为什么减：

- 减少下载成本
- 减少解析和执行成本
- 缩短首屏阻塞时间

---

## Q20. 性能优化最重要的方法论是什么？

**答：**

我认为最重要的方法论是：不要凭感觉乱优化，而是建立“度量 -> 定位 -> 优化 -> 验证 -> 监控”的闭环。

具体就是：

1. 明确用户痛点
2. 采集指标
3. 找到真正瓶颈
4. 针对性优化
5. 验证效果
6. 建立长期监控

因为很多所谓“性能优化技巧”如果脱离场景，可能只是增加复杂度，并没有真正改善用户体验。
